Metadata Property-Level Reference

Source-derived inventory for Controller screen metadata and linked runtime domains.

Contract boundary

This records the inspected C# models; it does not claim every property is a stable public standard. The central screen object is DynamicScreenSchema. Workflow, dashboard and report definitions can be embedded or linked while retaining separate lifecycles.

11. API, security, worker and scheduler findings

DomainFinding
APIActions/rules describe endpoint calls and runtime code composes CRUD routes. No single versioned Controller API compiler model was established.
SecurityRole hints occur in catalog, actions, workflow, dashboards and post-save rules, but no canonical inheritance model spans UI/API/data/report/tool execution.
WorkerNo canonical Worker schema found. RunInBackground is a flag, not a queue/lease/retry contract.
SchedulerNo canonical Schedule schema found. Workflow timeouts and dashboard refresh intervals are local timing metadata, not a Scheduler Engine.

12. Serializer and JSON compatibility

ConcernSource-derived behavior
Central read pathLoadSchemaAsync calls EncodingUtility.ReadJsonFileAsync<DynamicScreenSchema>.
Read casingThe utility defaults to PropertyNameCaseInsensitive = true, tolerating PascalCase/camelCase matching.
Unknown propertiesNo strict unmapped-member rejection is configured in the central utility; default deserialization does not make unknown fields errors.
Write casingNot uniform: onboarding/catalog writers use default naming while Layout Designer and other paths explicitly use camelCase.
Special shapesDashboard layout uses a custom converter supporting legacy row-array and newer layout-object forms.
ConsequenceJSON is permissive compatibility metadata, not one enforced versioned JSON Schema.

See ADR-0007.

13. Code On Time conversion boundary

The lightweight CotModels.cs retains only Controller Name, Fields, Views; field Name, Label, Type, DataController, DataView, DataFields; view Id, DataFields; and view-field FieldName. It cannot represent all observed legacy XML commands, categories, actions, roles, rules and reports. See Conversion Fidelity.

Architecture and roadmap

Catalog identity → DynamicScreenSchema → fields / sections / views / lookups → UI actions / pre-save / persistence / post-save → embedded or linked workflow → profile / dashboard / report / tools
  1. Define a versioned Controller Intermediate Representation.
  2. Publish JSON Schema only after casing, defaults, compatibility and unknown-property behavior are verified.
  3. Separate authored metadata from compiled/runtime metadata.
  4. Unify scope, policy and provenance across linked domains.
  5. Add Worker/Scheduler schemas only with owned runtimes and failure semantics.
