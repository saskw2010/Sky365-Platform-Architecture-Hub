Supporting, Background and Missing Engine Contracts

Independent evidence cards for capabilities previously grouped under one “not verified” catalog row.

Corrected classification

Source inspection found specific workers, audit services, workflow notifications and local file storage. These are real executable components, but only Audit approaches a cross-cutting Engine boundary. Scheduler, Plugin/Extension lifecycle and Vector Sync remain unestablished in the inspected current platform.

1. Worker Engine

Status: component workers exist; general Engine not established.

Purpose
Run durable metadata-defined background work independently of an interactive request.
Existing responsibilities
MetadataGraphWarmupService warms metadata at host startup; ProviderHealthMonitor performs provider health monitoring.
Inputs → outputs
Host lifecycle/configuration → warmed graph or provider health state.
Runtime
IHostedService and BackgroundService, registered through ServiceCollectionExtensions.
Metadata used
Service-specific configuration; no common Worker definition schema.
Dependencies
Host DI, metadata graph or provider services, logging.
Extension points
Normal DI registration only; no governed worker registry.
Gap / failure semantics
No shared queue, lease, retry, idempotency, cancellation history, tenant scope or Controller ownership contract.

2. Scheduler Engine

Status: not established.

Purpose
Trigger owned jobs using governed schedules.
Responsibilities
Schedule parsing, timezone and misfire behavior, concurrency locks, pause/resume, history and monitoring.
Inputs → outputs
Schedule + job identity + scope → execution request and durable outcome.
Runtime / evidence
No complete scheduler boundary or runtime registration was found. Mentions of Quartz/Hangfire occur in audit documentation, not executable platform integration.
Metadata used
No canonical schedule metadata found.
Dependencies
Future Worker Engine, durable store, clock, locking, audit.
Extension points
Future job-handler contract.

3. Audit and Flight Recorder Engine

Status: advanced partial.

Purpose
Record agent decisions, policies, tools, operational events and evidence for review.
Responsibilities
Trace capture, audit summaries, metrics, persistent or in-memory mascot audit and observability queries.
Inputs → outputs
Agent/tool/policy/memory events → trace records, summaries, metrics and dashboard data.
Runtime
AgentTraceService implements audit, observability, metrics, tool, memory and policy trace interfaces; PersistentMascotAuditService supplies persistence.
Metadata used
Trace and audit contracts, request/user/session/tool/policy context.
Dependencies
Agent runtime, persistence, logging, Agent Observatory API/UI.
Extension points
Audit/observability interfaces and alternate persistent/in-memory implementations.
Gap
Not a universal Controller/UI/CRUD/workflow/report audit bus; cross-engine replay and retention governance remain incomplete.

4. Notification Engine

Status: partial subsystem.

Purpose
Deliver workflow and domain notifications.
Responsibilities
Transform workflow notification events into delivery attempts; domain notification models/services exist elsewhere.
Inputs → outputs
WorkflowNotificationEvent → dispatched/logged notification action.
Runtime
WorkflowNotificationDispatcher invoked by WorkflowEngineService.
Metadata used
Workflow transition/notification definitions and event context.
Dependencies
Workflow runtime, user/recipient resolution and logging.
Extension points
IWorkflowNotificationDispatcher.
Gap
No unified channel/provider/template/preference registry, delivery lifecycle, deduplication or platform retry policy.

5. File and Media Engine

Status: local storage subsystem.

Purpose
Store, retrieve and delete uploaded form files.
Responsibilities
Generate names, write local files, expose URLs/streams and delete files.
Inputs → outputs
IBrowserFile/IFormFile → stored filename, URL or stream.
Runtime
Modules/FormBuilder/Infrastructure/Storage/FileStorageService.cs, rooted under wwwroot/uploads.
Metadata used
No platform file metadata contract; filenames and path configuration only.
Dependencies
Local filesystem, ASP.NET upload types and logging.
Extension points
IFileStorageService.
Gap
No provider abstraction, Controller attachment schema, ACL, antivirus scanning, versioning, lifecycle policy, transformation pipeline or content audit.

6. Plugin Engine

Status: not established.

Purpose
Discover, install and govern independently packaged platform capabilities.
Inputs → outputs
Signed manifest/package → isolated compatible extension registration.
Runtime / evidence
No plugin manifest, installer, compatibility resolver or sandbox lifecycle found in the inspected current platform.
Metadata used
None verified.
Dependencies
Future package trust, versioning, permission and isolation services.
Extension points
Future plugin contracts; ordinary DI is not a Plugin Engine.

7. Extension Engine

Status: architectural extension points only.

Purpose
Allow governed replacement or augmentation of Controller/runtime behavior.
Existing foundation
DI interfaces, renderer components, workflow notifier, file storage and integration adapters are replaceable seams.
Inputs → outputs
Implementation registration → substituted service behavior.
Runtime / metadata
Compile/startup DI; no extension manifest or runtime discovery metadata.
Dependencies
Host DI and each owning subsystem.
Gap
No compatibility, isolation, permissions, enable/disable, upgrade, rollback or failure-containment lifecycle.

8. Vector Sync Engine

Status: not established.

Purpose
Synchronize versioned, permission-aware source and Controller content into vector indexes.
Responsibilities
Chunk identity, embeddings, incremental upsert/delete, ACL/provenance propagation, retry, drift and metrics.
Inputs → outputs
Versioned documents/metadata + embedding template → indexed vectors and sync ledger.
Runtime / evidence
No executable embedding provider, vector store adapter or sync pipeline found. Strategy/research mentions are not runtime evidence.
Metadata used
Future source, chunk, embedding, ACL, tenant and Controller identity contracts.
Dependencies
Stable scoped identity, provenance registry, policy filters, vector provider and evaluation.
Extension points
Future embedding/vector adapters and source connectors.

Controller and Code On Time implications

  • A future Worker definition should reference immutable system/module/schema/profile identity rather than a display Controller name.
  • Scheduler must trigger typed Controller commands, not arbitrary screen URLs.
  • Audit must capture Controller identity, view/action, record keys, policy decision and conversion provenance.
  • Notifications can derive from workflow/action metadata, while file attachments require explicit field/record ownership metadata.
  • Plugins/extensions must not silently mutate converted Code On Time semantics.
  • Vector Sync must preserve Controller/field identity, ACL and deletion events.
