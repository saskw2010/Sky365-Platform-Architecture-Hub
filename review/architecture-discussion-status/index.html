1. هوية المنصة والاستيراد — تم

ناقشنا

  • هل SKY365 ERP أم Platform؟
  • مكان Code On Time: مرجع أم Runtime dependency؟
  • Converter/Import من COT وRadzen/Evolutility وقواعد البيانات.

قررنا

  • SKY365 هي Metadata-Driven Application Platform؛ الـERP تطبيق فوقها.
  • Code On Time external reference وbaseline للمقارنة، وليس مصدر الحقيقة.
  • Import from Other Platforms Engine قدرة دائمة وليست عملية migration لمرة واحدة.
  • نستفيد من أفكار Evolutility في Group/Layout/Component composition، ولا ننقل React runtime.

الباقي

  • إثبات adapters الفعلية لكل منصة من المصدر.
  • تعريف Import Fidelity Report وconversion-loss contract.

2. Controller Aggregate وScreen Studio — تم القرار، التنفيذ لاحقًا

ناقشنا

  • هل Screen Studio مولد كود أم محرر Metadata؟
  • هل نقسم Controller وScreen إلى مصادر مستقلة؟
  • Views متعددة، fields، layout، actions، permissions، query، charts وbuilders.
  • مكان الحقيقة، versioning، JSON snapshot والـRuntime.

قررنا

  • لا نغير المسار الحالي؛ نطوره.
  • Controller واحد يمكن أن يحتوي كل التعريف وهو Aggregate في قاعدة البيانات.
  • الـSections داخل Controller لها حدود واضحة للـBuilders، لكنها ليست مصادر حقيقة منفصلة.
  • Screen Studio والـBuilders تعدّل metadata ولا تولّد Components مخصصة.
  • كل Save ينشئ Draft Revision؛ لا يؤثر في المستخدمين.
  • Preview يختبر الـDraft، وPublish فقط يحدد Published Version.
  • Publish يصدر JSON Runtime Snapshot؛ الـRuntime يقرأ آخر Published فقط.
  • Database هي source of truth؛ JSON/cache للسرعة؛ Rollback يعيد نشر نسخة سابقة.

ثبت من الكود

  • System.SysControllers وFields وVersions موجودة.
  • Snapshot DB وRevert موجودان.
  • Runtime VGem3 يقرأ JSON file، والحلقة بين DB publish والـRuntime غير مكتملة.
  • يوجد سلوكان مختلفان في خدمتي Controller Config ويحتاجان توحيدًا لاحقًا.

الباقي داخل Goal 1

  • تحديد lifecycle states والـapproval actors والـconcurrency والـhash/cache invalidation بالتفصيل.
  • تثبيت ملكية كل Builder للـSections.
  • تعريف Publisher contract، دون تنفيذه الآن.

المراجع: الرؤية · ADR-0008 · فحص المصدر.

3. UI Engine وTheme وGroup — قرار التصنيف تم

قررنا

  • UI Engine محرك مستقل.
  • Theme ليس Engine منفصلًا الآن؛ هو tokens/variables وخدمة داخل UI Engine.
  • Group داخل مخطط الشاشة لتحسين UX/UI، ويمكن أن يحتوي حقولًا أو Component أو Master Component.
  • Group لا يعني Dashboard.
  • Static generated tags وdynamic segments أفكار توافق/تركيب تحتاج عقدًا موحدًا.

الباقي

  • Group JSON schema، nesting، responsive layout، component registry، binding، preview parity وundo/redo.
  • تحديد متى تستحق Theme الانفصال مستقبلًا إذا صار لها lifecycle وخدمات مستقلة.

4. SQL / Query / Data Source — البند التالي

ناقشنا ووثقنا

  • Controller في COT يحتوي Commands ويربط View بـCommandId.
  • العينة المفحوصة: 32 SELECT commands و44 view-command relations؛ Converter الحالي يفقد هذه semantics.
  • Runtime الحالي يبني Auto Query من TableName + Fields + Lookups.
  • يوجد CustomSql/VisualQuery مسار منفصل، وEasyQuery مسار ثالث.
  • Query Designer يمكن أن يخدم Grid وReport وDashboard وBI، وليس Grid فقط.
  • Read Model يجب ألا يساوي Write Model؛ joined query قد يقرأ عدة جداول، بينما الحفظ يحتاج WriteMap/Commands.

اتجاهنا المبدئي

  • SKY365 تملك عقد SkyQueryPlan/DataSourceDefinition.
  • EasyQuery محرر بصري عبر Adapter، وليس مصدر الحقيقة.
  • SQL output مشتق ومحكوم، وليس العقد الأساسي.
  • Auto Query للـCRUD البسيط، Compiled Command للمصادر المعقدة القابلة لإعادة الاستخدام، وGoverned Raw SQL كحل legacy مؤقت.

القرار المطلوب الآن

اعتماد شكل Data Source داخل Controller: الهوية، الجداول والعلاقات، projection، filters، parameters، lookups، aggregation، security predicates، read-only/mutability، وWriteMap المنفصل.

أسئلة هذا البند

  1. هل DataSourceDefinition Section داخل Controller ويمكن إعادة استخدامه بالـID؟
  2. هل View تربط DataSourceRef فقط أم تملك overrides؟
  3. متى نسمح Raw SQL ومن يعتمدها؟
  4. كيف نمثل lookup وjoin مرة واحدة لكل Grid/Report/Dashboard؟
  5. كيف يحدد WriteMap حفظ master/detail والـbatch؟

5. Semantic / AI — لم يُغلق

ناقشنا

  • Semantic Mapper، dictionary/graph، WrenAI sidecar، Agents/Skills، RAG/Vector/MCP.
  • Wren للفهم الدلالي لا للصلاحيات أو التنفيذ؛ SKY365 Gateway يحتفظ بالTenant/Policy/Audit.
  • AI Screen Planner ينتج typed plan يمر بالvalidators والاعتماد البشري.

الباقي

  • مكان Semantic Mapper داخل Controller lifecycle وعلاقته بـDataSourceDefinition.
  • ما الموجود فعليًا، الجزئي والمفقود في AI readiness.
  • قرار Wren proof-of-concept من عدمه بعد تثبيت Query contract.

6. Workers / Scheduler — لم يُغلق

ناقشنا

  • Worker متعلق بالـController، وWorker Engine يكتشف ويشغل workers.
  • الفرق بين Definition/Builder/Runner/Runtime، وبين Worker وService وSchedule.

الباقي

  • Worker JSON contract، parameters، queue، retry، idempotency، schedule، ownership، monitoring وaudit.
  • إثبات الموجود فعليًا في الكود وتصنيف Worker Engine مقابل Scheduler Engine.

7. Dashboard / BI / Reports — لم يُغلق

ناقشنا

  • هل Dashboard داخل Controller أم JSON مستقل مع tags/references؟
  • Query/Data Source يجب أن يخدم Grid وReport وDashboard.
  • Metabase benchmark/optional BI portal، وليس قلب operational runtime.

الباقي

  • Dashboard definition ownership، widgets/blocks، bindings، permissions، drill-down وrefresh/cache.
  • الحد الفاصل بين Dashboard/BI Engine وReport Engine وVisualization داخل UI Engine.

8. Offline / Batch Saving — لم يُغلق

ناقشنا

  • POS، master/detail batch، local buffering، SQLite/local storage/Redis، submit واحد، sync/conflict.
  • VGem3 يحتوي Buffered persistence/local-storage draft foundation.

الباقي

  • Offline Engine أم Sync subsystem؟
  • Outbox، local database، encryption، conflict policy، retry، reconciliation والـtenant boundaries.

9. Engine Catalog — Baseline فقط

تم اعتماد قاعدة التصنيف: ليس كل Service محركًا. المحرك يحتاج Purpose وعقود Metadata وRuntime lifecycle وخدمات وextension points واضحة. القائمة الحالية 27 مرشحًا، لكن ما زال مطلوبًا فحص كل محرك بندًا بندًا وإعادة العدد النهائي بعد إزالة الأسماء التسويقية أو دمج subsystems.
