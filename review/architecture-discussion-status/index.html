1. هوية المنصة والاستيراد — تم

ناقشنا

  • هل SKY365 ERP أم Platform؟
  • مكان Code On Time: مرجع أم Runtime dependency؟
  • Converter/Import من COT وRadzen/Evolutility وقواعد البيانات.

قررنا

  • SKY365 هي Metadata-Driven Application Platform؛ الـERP تطبيق فوقها.
  • Code On Time external reference وbaseline للمقارنة، وليس مصدر الحقيقة.
  • Import from Other Platforms Engine قدرة دائمة وليست عملية migration لمرة واحدة.
  • نستفيد من أفكار Evolutility في Group/Layout/Component composition، ولا ننقل React runtime.

الباقي

  • إثبات adapters الفعلية لكل منصة من المصدر.
  • تعريف Import Fidelity Report وconversion-loss contract.

2. Controller Aggregate وScreen Studio — تم القرار، التنفيذ لاحقًا

ناقشنا

  • هل Screen Studio مولد كود أم محرر Metadata؟
  • هل نقسم Controller وScreen إلى مصادر مستقلة؟
  • Views متعددة، fields، layout، actions، permissions، query، charts وbuilders.
  • مكان الحقيقة، versioning، JSON snapshot والـRuntime.

قررنا

  • لا نغير المسار الحالي؛ نطوره.
  • Controller واحد يمكن أن يحتوي كل التعريف وهو Aggregate في قاعدة البيانات.
  • الـSections داخل Controller لها حدود واضحة للـBuilders، لكنها ليست مصادر حقيقة منفصلة.
  • Screen Studio والـBuilders تعدّل metadata ولا تولّد Components مخصصة.
  • كل Save ينشئ Draft Revision؛ لا يؤثر في المستخدمين.
  • Preview يختبر الـDraft، وPublish فقط يحدد Published Version.
  • Publish يصدر JSON Runtime Snapshot؛ الـRuntime يقرأ آخر Published فقط.
  • Database هي source of truth؛ JSON/cache للسرعة؛ Rollback يعيد نشر نسخة سابقة.

ثبت من الكود

  • System.SysControllers وFields وVersions موجودة.
  • Snapshot DB وRevert موجودان.
  • Runtime VGem3 يقرأ JSON file، والحلقة بين DB publish والـRuntime غير مكتملة.
  • يوجد سلوكان مختلفان في خدمتي Controller Config ويحتاجان توحيدًا لاحقًا.

الباقي داخل Goal 1

  • تحديد lifecycle states والـapproval actors والـconcurrency والـhash/cache invalidation بالتفصيل.
  • تثبيت ملكية كل Builder للـSections.
  • تعريف Publisher contract، دون تنفيذه الآن.

المراجع: الرؤية · ADR-0008 · فحص المصدر.

3. Persistence Engine — مؤجل

طلب المالك تأجيل هذا المحرك لأنه يحتاج جلسة مستقلة وهادئة. حُفظت الموضوعات التالية دون حسم:

  • Immediate وAtomic وBuffered/Hybrid persistence.
  • Master/Detail وgenerated keys وترتيب عمليات الحفظ.
  • Insert/Update/Delete وWriteMap/Commands.
  • Batch saving وSubmit واحد.
  • Transactions وrollback وpartial failure.
  • Concurrency وidempotency وretry.
  • Validation وbusiness rules قبل/بعد الحفظ.
  • Offline local store/outbox/sync/conflict reconciliation.
  • Security، tenant scope، audit والأداء.

الحالة: مؤجل؛ لا قرار معماري جديد.

4. UI Engine وTheme وGroup — التالي الآن

قررنا سابقًا

  • UI Engine محرك مستقل.
  • Theme ليس Engine منفصلًا الآن؛ هو tokens/variables وخدمة داخل UI Engine.
  • Group داخل مخطط الشاشة لتحسين UX/UI، ويمكن أن يحتوي حقولًا أو Component أو Master Component.
  • Group لا يعني Dashboard.
  • Static generated tags وdynamic segments أفكار توافق/تركيب تحتاج عقدًا موحدًا.

الباقي لمراجعة المحرك

  • Purpose، inputs/outputs، runtime boundary والخدمات المالكة.
  • Component Registry وDynamicComponent resolution.
  • Grid/Form/Tabs/Groups/Charts/Query Builder وMaster Components.
  • Responsive layout، binding، events، validation presentation وaccessibility.
  • Studio preview مقابل Runtime parity، undo/redo وschema migrations.
  • حدود UI Engine مع Controller Runtime وDashboard وWorkflow وAI.
  • Group JSON schema وnesting.
  • تحديد متى تستحق Theme الانفصال مستقبلًا إذا صار لها lifecycle وخدمات مستقلة.

4. البنود الباقية داخل Metadata / Controller Engine

تصحيح التصنيف: SQL/Query ليس مرحلة رئيسية خارج المحرك الحالي؛ هو capability داخل عقد الـController. كذلك Builder وAI وSemantic وSync نقاط داخل دورة تأليف ونشر الـmetadata، مع وجود تقاطعات لاحقة مع محركات UI وAI وDashboard.

البند الداخليما ناقشناهما بقي
Query / Data Source MetadataCOT Commands، Auto Query، CustomSql، EasyQuery، lookup/join، وتقاسم المصدر بين Grid/Report/Dashboard.اعتماد DataSourceDefinition/SkyQueryPlan وRead Model/WriteMap وحدود Raw SQL.
Screen Studio BuildersGrid/Form/Query/Chart/Actions/Access builders تعدل Sections داخل Controller DB Aggregate.ملكية كل Section، typed commands، concurrency، validation وpreview contract.
UI metadataViews، fields، groups، tabs، layout، actions، permissions وtheme inputs مشتقة من Controller.View-specific presentation وcomponent registry والعلاقة الرسمية مع UI Engine.
AI query/screen generationAI يولد typed plan/query ثم validation وhuman approval؛ لا يكتب SQL أو JSON حرًا للتشغيل.Skills contracts، approval، diagnostics، وكيف يحفظ الناتج Draft Revision.
Semantic Dynamic MapperMapping بين لغة العمل وentities/fields/lookups/relations/metrics، وربطه بالـController والـAI query.تحديد مكان الـmapping metadata، lifecycle، versioning، وعلاقة ZEVAC التي ذكرها المالك بعد التحقق من الاسم والمصدر.
Metadata Sync / PublishDB source of truth، Draft/Preview/Publish، وتصدير JSON Snapshot للسرعة.Publisher، hashes، cache invalidation، dependency sync، rollback وتزامن semantic mappings.

حالة المحرك

مؤجل بطلب المالك بعد حفظ كل البنود. لن نستكمل Query/Data Source أو Semantic Mapper الآن.

5. Controller Runtime + CRUD Engine — مؤجل بعد حفظ القصص

هذا المحرك Verified كحد Runtime مبدئيًا، لكن مراجعته التفصيلية مؤجلة. البنود التالية محفوظة حتى نعود إليه:

المجموعةنقاط المراجعة المؤجلة
ثلاثة مسارات Runtimeمقارنة Classic Screen وVGem3 وVGPT5: اختيار schema، تحميلها، بناء query، تحميل rows/lookups، rendering، actions، save، errors وperformance.
بناء Queryكيف يبني كل Runtime الـSELECT والـprojection والـfilters والـlookups والـjoins والـsorting/paging؛ وهل يستخدم Auto Query أو CustomSql أو EasyQuery.
Screen lifecycleResolve Controller/Profile → load published snapshot → authorize → build screen/context → load data → interact → validate → action/save → refresh/audit.
CRUDCreate/Read/Update/Delete، source mutability، primary keys، defaults، validation، concurrency، master/detail وtransactions.
Components داخل ScreenGrid، Form، Lookup، Tabs/Groups، Chart، Query Builder، Dashboard blocks وMaster Components، وكيف تستقبل metadata/context.
Dashboard داخل Screenهل هو View/Component binding داخل الشاشة أم انتقال إلى Dashboard Engine، وما contract البيانات والصلاحيات والrefresh/drill-down.
Workflow داخل Screenعرض الحالة والـtasks/history، الأزرار المتاحة، Start Workflow، transition/approve/reject، وربط record/controller/workflow definition.
Workers داخل Screenعرض حالة worker أو تشغيل action يطلب job؛ الشاشة لا تنفذ worker بنفسها، ويجب توثيق boundary مع Worker Engine.
ReviewsReview/approval components، التعليقات، الحالة، history، الصلاحيات وعلاقتها بالWorkflow/Audit.
ActionsNew/Edit/Delete/Save/Export/Approve/Start Workflow/custom actions، visible/enabled conditions، permissions وserver/client execution.
SecurityScreen/View/Field/Action/Record/Query permissions، tenant scope، وعدم الاعتماد على إخفاء UI وحده.
Runtime contextUser/tenant/system/module/controller/profile/mode/record/parameters، وتمرير السياق لكل component ومحرك متكامل.
API/Audit/Errorsالعلاقة مع Dynamic API، diagnostics، trace IDs، audit events، failure handling وfallback behavior.

حالة المحرك

مؤجل بطلب المالك. لا قرار جديد ولا تنفيذ. عند العودة نبدأ بمقارنة evidence table بين Classic وVGem3 وVGPT5 بدل إعادة الحوار من البداية.

5. Semantic / AI — لم يُغلق

ناقشنا

  • Semantic Mapper، dictionary/graph، WrenAI sidecar، Agents/Skills، RAG/Vector/MCP.
  • Wren للفهم الدلالي لا للصلاحيات أو التنفيذ؛ SKY365 Gateway يحتفظ بالTenant/Policy/Audit.
  • AI Screen Planner ينتج typed plan يمر بالvalidators والاعتماد البشري.

الباقي

  • مكان Semantic Mapper داخل Controller lifecycle وعلاقته بـDataSourceDefinition.
  • ما الموجود فعليًا، الجزئي والمفقود في AI readiness.
  • قرار Wren proof-of-concept من عدمه بعد تثبيت Query contract.

6. Workers / Scheduler — لم يُغلق

ناقشنا

  • Worker متعلق بالـController، وWorker Engine يكتشف ويشغل workers.
  • الفرق بين Definition/Builder/Runner/Runtime، وبين Worker وService وSchedule.

الباقي

  • Worker JSON contract، parameters، queue، retry، idempotency، schedule، ownership، monitoring وaudit.
  • إثبات الموجود فعليًا في الكود وتصنيف Worker Engine مقابل Scheduler Engine.

7. Dashboard / BI / Reports — لم يُغلق

ناقشنا

  • هل Dashboard داخل Controller أم JSON مستقل مع tags/references؟
  • Query/Data Source يجب أن يخدم Grid وReport وDashboard.
  • Metabase benchmark/optional BI portal، وليس قلب operational runtime.

الباقي

  • Dashboard definition ownership، widgets/blocks، bindings، permissions، drill-down وrefresh/cache.
  • الحد الفاصل بين Dashboard/BI Engine وReport Engine وVisualization داخل UI Engine.

8. Offline / Batch Saving — لم يُغلق

ناقشنا

  • POS، master/detail batch، local buffering، SQLite/local storage/Redis، submit واحد، sync/conflict.
  • VGem3 يحتوي Buffered persistence/local-storage draft foundation.

الباقي

  • Offline Engine أم Sync subsystem؟
  • Outbox، local database، encryption، conflict policy، retry، reconciliation والـtenant boundaries.

9. Engine Catalog — Baseline فقط

تم اعتماد قاعدة التصنيف: ليس كل Service محركًا. المحرك يحتاج Purpose وعقود Metadata وRuntime lifecycle وخدمات وextension points واضحة. القائمة الحالية 27 مرشحًا، لكن ما زال مطلوبًا فحص كل محرك بندًا بندًا وإعادة العدد النهائي بعد إزالة الأسماء التسويقية أو دمج subsystems.
