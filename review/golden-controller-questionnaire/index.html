توسعة Q001 — القصة كما شرحها مالك المنصة

الوضع الحاليSKY365 يملك Converter Engine يستطيع قراءة Controller من منصة خارجية مثل Code On Time وتحويله إلى صيغة SKY365 الداخلية. Code On Time ليس حالة وحيدة؛ هو أول مثال ضمن نظم خارجية متعددة يمكن تحويلها مستقبلًا.
الطبيعي / المرجعينستخدم إمكانيات النظام الخارجي كـbaseline للمقارنة: ما الذي يدعمه Controller الخارجي، وما الذي يغطيه Controller الداخلي، وما التحسينات التي تضيفها SKY365 فوقه.
الهدفاستيراد ERP مبني على Code On Time وتحويل Controllers مباشرة إلى SKY365، ثم تشغيلها داخل محركات SKY365 دون اعتماد تشغيلي على Code On Time.
ملاحظة مكتشفةController الخاص بـCode On Time يحتوي SQL Commands، بينما مخرجات/نموذج الـConverter الحالي لا يظهر فيه تمثيل واضح لهذه الـCommands.
التصنيف الآناستفسار مفتوح، وليس إيجابية أو سلبية بعد. يلزم فحص أنواع الـCommands، ما إذا كانت مطلوبة وقت التشغيل، وكيف ينبغي تمثيلها وتأمينها داخل SKY365.

Q001-A — سياق Import From Other Platform Engine

تصحيح الاسم والنطاقالقدرة المقصودة ليست Converter خاصًا بـCode On Time. الاسم المعماري الذي حدده مالك المنصة هو Import From Other Platform Engine، ويُفترض أن يستوعب عدة source adapters.
الوضع الحالي حسب المالكتمت كتابة قائمة بالأدوات/المنصات المراد الاستيراد منها: Code On Time وRadzen وObout، بالإضافة إلى Import From Database Schema.
الاسم الثالث المثبتObout. تم تثبيته من صفحة الرؤية الداخلية sky365-studio-pitch.html، وليس «Optility».
الرؤية المكتوبةالصفحة تسمي القدرة Multi-Source Import Engine وتحدد المصادر: CodeOnTime + Radzen + Obout + SQL direct، وتصف الناتج بأنه JSON Schema موحد، مع حالة «قيد التطوير».
الوضع المثبت من الكوديوجد مسار فعلي لاستيراد Code On Time XML، ومسار لتوليد SysController من جدول/عرض في قاعدة البيانات. يوجد _templates/RadzenMigration كقوالب هجرة، لكن لم يثبت فحص هذه الجولة وجود Adapter موحد يشغّل استيراد Radzen أو Obout إلى canonical schema.
Database Schema الحاليGenerateFromDatabaseTableAsync يقرأ schema عبر اتصال EF، لكنه يستخدم صيغة SQL Server المحددة SELECT TOP 0 ... وbracket quoting. لذلك التطبيق المثبت حاليًا SQL Server-oriented.
الطبيعي / المرجعيكل منصة خارجية أو نوع قاعدة بيانات يكون Source Adapter مستقلًا، ثم يحول نموذجه إلى Canonical SKY365 Controller Model واحد بدل إدخال منطق كل منصة داخل الـRuntime.
الهدفدعم استيراد تطبيقات/Controllers من منصات خارجية متعددة، ودعم schema discovery من عدة أنواع قواعد بيانات، مع أن أول implementation لقاعدة البيانات هو SQL Server.
أثره على Q002قرار SQL Commands يجب أن يكون جزءًا من canonical import contract وسياسة capabilities، لا استثناءً خاصًا بـCOT. كل Adapter يعلن ما اكتشفه وما حوّله وما لم يدعمه.
أسئلة مفتوحةتحديد Contract كل Adapter؛ تحديد أنواع قواعد البيانات التالية بعد SQL Server؛ وتحديد شكل مدخل Radzen وObout: metadata/project files أم تحليل generated code/UI.

Q001-B — React Visual Builder Comparator

الحالةقصة المالك مسجلة؛ اسم المنتج لم يُثبت بعد.
الوضع المرجعيتوجد منصة مقارنة تبني الشاشات من JSON/metadata، وتمتاز بوجود Visual Layout UI، بينما renderer/model الخاص بها مبني على React.
ملاحظة السوقمنظومة أدوات AI وvibe coding تنتج React بسرعة بسبب اتساع ecosystem والقوالب والبيانات التدريبية المتاحة؛ هذه أفضلية سرعة tooling وليست حكمًا بأن React هو النموذج المعماري الصحيح لـSKY365.
موقف المالكالمالك لا يريد تحويل SKY365 إلى React؛ يريد بناء الـUI والـVisual Designer باستخدام Blazor، مع ناتج منظم داخل .NET assemblies ومنظومة تقنية موحدة.
الهدفإغلاق فجوة Visual Layout Designer وسرعة التوليد داخل Blazor، مع الحفاظ على Metadata-Driven Runtime وJSON definitions، بدل تغيير الـruntime إلى React فقط لأن أدوات AI تولده أسرع.
تصحيح أمنيالأمان لا ينتج تلقائيًا من React أو Blazor. أي كود يعمل في المتصفح — JavaScript أو WebAssembly assembly — يجب اعتباره غير موثوق. صلاحيات البيانات والتنفيذ والتحقق من المدخلات يجب فرضها على الخادم. أفضلية Blazor هنا هي توحيد .NET والنماذج والأنواع وقابلية الحوكمة، وليست سرية assembly في المتصفح.
قرار مبدئيلا React pivot. يُعامل المنتج الخارجي كمرجع capabilities للـlayout/design experience، بينما يبقى Target Renderer الخاص بـSKY365 هو Blazor ما لم يصدر قرار معماري لاحق يغير ذلك.
استفسار مفتوحتثبيت اسم/رابط المنتج المقصود، ثم مقارنة JSON model وlayout designer وcomponent registry وruntime معه مقارنة مباشرة.

Q006 — قاعدة تصنيف المحركات والإحصاء الأولي

هل مهمة اكتشاف كل المحركات انتهت؟لا. المكتمل حتى الآن مراجعة مركزة حول Controller وConverter وGroup وبعض مسارات الـDynamic Runtime. لا يوجد بعد Engine Catalog نهائي يغطي النظام كله.
المثبت باسم Engine في كود التشغيلسبعة implementations أولية: WorkflowEngineService، PivotAnalyticsEngine، DynamicBusinessRuleEngine، DynamicIntegrationEngine، DynamicPersistenceEngine، DynamicReportFilterEngine، وMascotPolicyEngineService. هذا ليس العدد النهائي لأن بعض المحركات قد تحمل اسم Runtime/Service، وبعض الأسماء Engine قد تكون صفحة UI فقط.
أجزاء Runtime قوية مرشحة للتصنيفSky365DynamicRuntimeService وVisualQueryRuntimeService ومسارات Runtime VGem3/VGpt5/Index. يلزم تحليل المسؤوليات قبل عدّها كمحركات مستقلة أو runtime profiles لمحرك واحد.
قرار UI + Themeبند ومحرك واحد: UI Engine. الـTheme والألوان Design Tokens/variables وSubsystem تدخل إلى UI Engine وتغيّر طريقة الرسم؛ ليست Engine مستقلة في التصنيف الحالي. الدليل التنفيذي المساعد هو ThemeCssBuilder. يعاد فتح الفصل فقط إذا ظهر مستقبلًا lifecycle ونشر وإصدارات وعقد تشغيل مستقل للـTheme.
علاقة Engine بالـServicesالمحرك Capability وحدود تشغيل، وليس شرطًا أن يساوي Class واحدة أو Service واحدة. قد يديره Orchestrator Service ومعه عدة خدمات متخصصة وBuilders وRunners وRepositories. وقد تشارك خدمة بنية تحتية — مثل metadata أو security — بين عدة محركات. المطلوب منع Service واحدة عملاقة تدير النظام كله ومنع تسمية كل Service بأنها Engine.
شرط العد النهائيلكل مرشح نسجل: purpose، metadata/input، output، runtime entry point، services، dependencies، lifecycle، extension points، ودليل المصدر؛ ثم نصنفه Confirmed Engine أو Engine Subsystem أو Supporting Service/Builder أو UI Page فقط.

Q002/Q007 — Query Relations، Batch Saving وOffline

قصة المالكاستيراد SQL Command وبناء العلاقات منه أو بصريًا؛ تحديد الجدول الرئيسي المسؤول عن الحفظ؛ دعم Master/Detail متعدد الجداول؛ تجميع التغييرات ثم Submit واحد؛ وإتاحة Local-first persistence لتطبيقات مثل POS.
الموجود فعليًاDynamicScreenSchema.PersistenceModel يدعم Atomic وImmediate وHybrid وBuffered. DynamicPersistenceEngine يحفظ Master + Details ويملك Batch details داخل Transaction في Atomic، بينما VGem3 وVGpt5 يحفظان Buffered drafts في localStorage. يوجد Desktop offline shell وSQLite استخدامات منفصلة، لكن لا يوجد Sync Engine عام.
التقسيم المعماري المقترحQuery/Relation Definition للقراءة والـjoins؛ Aggregate Write Map يحدد Root Table والـDetails والمفاتيح وترتيب الحفظ؛ Persistence Engine ينفذ Unit of Work/Batch؛ وOffline/Sync Engine يدير local durable store وoutbox والمزامنة والتعارضات. لا نجعل SELECT join نفسه عقدة كتابة تلقائية.
حكم SQL الخاميُحفظ SQL المستورد كـprovenance/reference أو escape hatch محكوم، بينما العقدة المعتمدة للتشغيل تكون Query metadata وعلاقات وparameters وسياسة صلاحيات. الكتابة لا تُستنتج من SQL عرض متعدد الجداول؛ تُعرّف صراحة في Write Map.
اختيار التخزين المحليlocalStorage مناسب لمسودة صغيرة، وليس سجل معاملات POS. في Desktop يكون SQLite مرشحًا طبيعيًا. في browser/PWA يكون IndexedDB أو SQLite/WASM بطبقة معتمدة. Redis خدمة شبكة/cache وقد يفيد server-side staging، لكنه ليس مخزن Offline على جهاز العميل ولا المصدر الدائم للمعاملة.
ما ينقص Offline EngineLocal schema/versioning، transaction log/outbox، temporary IDs، dependency ordering، idempotency keys، retry/backoff، sync cursor، conflict detection/resolution، inventory/payment safety rules، encryption، user/device ownership، audit وrecovery.
المنافس المثبت داخل المشروعKorzh EasyQuery. QueryDesignerPanel.razor يستدعي sky365.initEasyQuery ويعيد QueryJson + SQL؛ Program.cs يسجل AddEasyQuery/MapEasyQuery؛ والمشروع يراجع حزم Korzh.EasyQuery.* 7.4.1. EasyQuery.JS يدعم JavaScript وSPA بما فيها React وAngular وVue. المالك صحح نطاقه: ليس للجريد فقط، بل مرشح لتصميم Data Sources مشتركة للتقارير والـDashboard أيضًا.
فجوة النسخة الحاليةحزم .NET المثبتة هي 7.4.1، بينما App.razor يحمل JavaScript/CSS من CDN بإصدار 7.2.3. هذا mismatch يحتاج تحقق توافق قبل اعتبار التكامل جاهزًا، خصوصًا أن التوثيق القديم يسجل EasyQuery API mismatch.
المقارناتActive Query Builder مرجع قوي لتحليل SQL ثنائي الاتجاه، relation canvas، metadata offline والقيود على objects. EasyQuery هو المنافس الموجود بالفعل داخل SKY365. FireDAC Cached Updates مرجع لفكرة master/detail change log والعمل offline. اسم KH Croc/Croc Engine لم يثبت من البحث الحالي.
الحكم الحاليالفكرة ممكنة ومناسبة لـSKY365، والأساس الجزئي موجود. لا نبدأ Engine جديدًا من الصفر قبل توحيد Buffered + Batch الحاليين داخل عقد Persistence، ثم نضيف Offline/Sync boundary مستقلًا فوقهما.

Q002-A — كيف تُبنى Query الجريد في الـRuntime الحالي؟

مسار الاستدعاءVGem3 وVGpt5 وDynamicSchemaView تستدعي Sky365DynamicRuntimeService.BuildRowsAsync. بعد تحميل الـSchema، حالتا Centers وComplexes تمران عبر Services مخصصة؛ بقية الشاشات تمر عبر BuildRowsGenericAsync.
اختيار المصدرالمصدر جدول واحد فقط من TableSchema + TableName. الـRuntime يتحقق من وجود الجدول ويقرأ أعمدته من قاعدة البيانات.
اختيار أعمدة الجريديأخذ حقول schema.Fields التي تحمل InList=true، ويستبعد أي اسم غير موجود فعليًا في أعمدة الجدول، ثم يضيف IdField عند الحاجة.
SQL الناتجSELECT [InList columns] FROM [schema].[table]، ثم يضيف ListFilters المسموحة كـparameters، ويضيف البحث كـTRY_CONVERT(... nvarchar) LIKE @sN على الحقول، ثم ORDER BY Id DESC.
ما لا يستخدمه الجريدالمسار الرئيسي لا يقرأ QueryJson أو قسم Commands، ولا ينفذ CustomSql، ولا ينشئ JOIN أو aliases أو calculated columns. VisualQueryRuntimeService مسار منفصل ينفذ CustomSql الخام، لكنه غير موصول بتحميل الجريد الرئيسي.
مزايا الوضع الحاليبسيط؛ أسماء identifiers تمر عبر normalization؛ القيم في filters/search parameterized؛ ويتجنب طلب أعمدة غير موجودة. مناسب لـController بسيط مبني على جدول واحد.
القيودلا يدعم Query متعددة الجداول، ولا semantics الخاصة بـView/Command، ولا projection محسوبة؛ البحث يحول أعمدة كثيرة إلى نص وقد يضعف استخدام indexes؛ ولا يظهر في هذا المسار paging/limit، لذلك قد يحمل صفوفًا كثيرة. كما أن خلط field metadata الخاصة بالعرض مع query construction يحد من التطور.
الطريقة الأفضلنحتفظ بالمسار الحالي كـAuto Table Query fallback، ونضيف لكل View مرجعًا إلى QueryDefinition/Command. EasyQuery يصمم QueryJson والعلاقات؛ Query Compiler يتحقق منها ويولد SELECT parameterized مع allowlist وpaging/sort/filter policy؛ والـGrid Runtime ينفذ Compiled Query. يظل WriteMap منفصلًا للحفظ ولا يُستنتج من JOIN.
ترتيب المصدر المقترح1) View لها Command معتمد → compiled query؛ 2) View بلا Command → auto table query الحالي؛ 3) raw SQL → compatibility/provenance فقط وبسياسة صريحة، وليس المسار الافتراضي.
تصحيح نطاق EasyQueryEasyQuery لا يقتصر على تعديل Command الجريد. يُعامل كواجهة Data Source Designer تصمم Query/Relations وparameters وprojection مرة واحدة، ثم يمكن ربط التعريف نفسه بـGrid View أو Report Dataset أو Dashboard Widget/Data Source.
الحد المعماري المقترحData Source Designer Engine يملك authoring وQueryJson؛ Query Compiler/Runtime يملك validation والتنفيذ؛ ثم Grid وReport Engine وDashboard/BI Engine مستهلكون مستقلون. EasyQuery أداة التصميم الحالية داخل الحد الأول، ولا يجب أن تصبح dependency داخل كل renderer.
نموذج المشاركةDataSourceDefinition لها ID وإصدار وConnectionRef وQueryDefinition وParameters وOutputSchema وSecurityPolicy. المستهلك يربطها عبر DataSourceRef ويضيف bindings/filters الخاصة به دون نسخ SQL.
ما لا يملكه Data Source Designerلا يقرر شكل Grid، ولا Report layout، ولا Dashboard widgets، ولا WriteMap. هو ينتج Dataset contract؛ وكل Engine مستهلك مسؤول عن العرض والصلاحيات الخاصة به.
قرار المالك المسجلEasyQuery يصمم Commands وQueries وRelations، ويمكن استخدامه كمحرك/مصمم Data Sources للتقارير والـDashboard بجانب شاشات الـRuntime.

Q002-B — كيف يظهر الـLookup في VGem3 حاليًا؟

الإجابة المباشرةSQL الجريد لا تعمل JOIN للـLookup. الجريد تجلب قيمة الـFK من الجدول الأساسي، ثم VGem3 يحمل قاموس خيارات الـLookup باستعلام منفصل ويحوّل الـID إلى Text عند العرض.
Query الجريدمثلًا: SELECT [Id], [CustomerId], ... FROM [dbo].[Orders]. الناتج يحتوي CustomerId=15 فقط.
Query الـLookupGetLookupOptionsAsync يحل Primary/legacy/Fallback source، ثم يولد تقريبًا: SELECT CAST(ValueField AS nvarchar) AS Value, CAST(TextExpression AS nvarchar) AS Text FROM LookupTable ORDER BY Text. يدعم DisplayTemplate أو TextField + SecondaryTextField للاسم الثنائي.
التحميلLoadLookupsAsync يحمل بالتوازي كل Lookups للحقول التي عليها InForm أو InList، ويخزنها في _lookupOptions[fieldName].
العرضGetRowDisplayValue يأخذ FK الخام من الصف، يبحث عن Option مساوية له، ثم يعرض option.Text. Filters داخل Grid تستخدم القيمة المعروضة، لذلك تظهر أسماء لا IDs.
الميزةفصل Lookup metadata عن Query الجدول؛ إعادة استخدام الخيارات في Form وGrid؛ ودعم مصدر Controller أو جدول مع fallback وقوالب عرض ثنائية.
المشكلاتتحميل قائمة Lookup كاملة لكل حقل؛ والبحث عن الاسم يتم بعد جلب البيانات وليس داخل SQL الرئيسية. لذلك البحث العام في الـRuntime يبحث في FK الخام لا اسم العميل، والفرز/الـpaging على النص المعروض لا يمكن دفعه بكفاءة لقاعدة البيانات. كذلك البحث الخطي عن Option لكل Cell قد يصبح مكلفًا مع جداول/Lookups كبيرة.
التصميم الأفضلHybrid Lookup Policy. للـForm والـLookups الصغيرة: نحتفظ بقائمة options مع cache/dictionary وlazy/paged loading. للـGrid والبحث والتقارير: Query Compiler يستخدم Relation metadata ويولد LEFT JOIN أو projection مع alias مثل CustomerId__Text، مع الاحتفاظ بـCustomerId للحفظ.
مثال الناتج الأفضلSELECT o.Id, o.CustomerId, c.Name AS CustomerId__Text FROM Orders o LEFT JOIN Customers c ON c.Id=o.CustomerId. الـUI يعرض CustomerId__Text، بينما WriteMap يحفظ CustomerId.
علاقة EasyQueryEasyQuery/Data Source Designer يستطيع توليد Relation والـprojection الخاصة بالـLookup للـGrid/Report/Dashboard. لكن Lookup field metadata تظل المصدر الدلالي للعلاقة، وQuery Compiler هو من يقرر SQL الآمنة.

القرار المقترح

  1. Command/EasyQuery Metadata هي المصدر الرسمي للـRead Model عندما تحتاج الشاشة Relation أو Lookup display أو تقريرًا أو Dashboard.
  2. Auto Table Query تبقى ولا نحذفها؛ يستخدمها الـController البسيط تلقائيًا عندما لا توجد Command مرتبطة بالـView.
  3. Custom SQL يُحفظ للاستيراد والتوافق، ولا ينفذ إلا كـSELECT داخل Policy: approved connection، allowlisted objects، parameters فقط، timeout، row limit، audit وقراءة فقط.
  4. الـRuntime لا يعتمد على EasyQuery كـvendor library؛ يعتمد على SKY365 DataSourceDefinition. EasyQuery أحد المصممين القادرين على إنتاجها.
  5. كل View تربط DataSourceRef/CommandId؛ Report وDashboard يعيدان استخدام المرجع نفسه، بينما WriteMap مستقل للحفظ.

قاعدة الاختيار: Simple CRUD → Auto Query؛ Joined/Reusable Dataset → Compiled Command؛ Imported unsupported legacy query → Governed Raw SQL مؤقتًا حتى يُحوّل إلى metadata.

Q002 — فحص الوضع الحالي من المصدر

الحالةتم إثبات الوضع الحالي؛ ننتظر قصة المالك ثم القرار.
الموجود في COT ReferenceTplMdTrsQuotation.xml يحتوي command1 من نوع Text. النص هو SELECT projection مع LEFT JOIN على العميل والمستودع، والـView grid1 يرتبط به من خلال commandId="command1".
ملاحظة المصطلحCommands في هذا المثال ليست بالضرورة INSERT/UPDATE/DELETE؛ الأمر المثبت هنا استعلام SELECT يحدد مصدر وشكل بيانات الـView.
Onboarding ConverterDynamicControllerOnboardingService يقرأ fields وخصائص lookup ويبني DynamicScreenSchema، لكنه لا يقرأ عقدة commands ولا يحتفظ بعلاقة view → commandId.
Runtime COT ParserCotDataController يمثل Fields وViews فقط ولا يحتوي Commands؛ لذلك CodeOnTimeXmlParser وMapCotToSchema لا يستطيعان نقلها في هذا المسار.
Designer Parserيوجد مسار منفصل في DesignerService يقرأ Command Id وType وText، لكنه ليس دليلًا أن الـDynamic onboarding JSON أو runtime يستخدم هذه المعلومات.
نتيجة المسح الأوسعفي 37 ملف COT مرجعي موجود داخل Salogi وNoor، تم العثور على 32 Command و44 علاقة view → commandId. جميع الـ32 كانت من النوع Text وتبدأ بـSELECT؛ لم يظهر في هذه العينة INSERT أو UPDATE أو DELETE أو EXEC.
معنى النتيجةالفجوة المثبتة في العينة الحالية هي فقدان Query/View semantics — مثل projection وjoins والحقول المحسوبة/المقروءة — وليست فقدان أوامر كتابة. لا يجوز تعميم ذلك تلقائيًا على كل نظام خارجي مستقبلي.
قدرة داخلية موجودةDynamicScreenSchema وDynamicViewSchema يحتويان خاصية CustomSql، كما توجد schemas تجريبية تحتفظ باستعلامات COT داخلها. لذلك يوجد مكان أولي، لكنه ليس عقد Query مكتملًا.
فجوة ربط الـRuntimeالمسار الرئيسي BuildRowsGenericAsync يتجاهل CustomSql ويبني SELECT من TableName + Fields. وDynamicSchemaView يستدعي هذا المسار، بينما يحقن IVisualQueryRuntimeService دون استخدام ظاهر في تحميل القائمة.
المسار التجريبيVisualQueryRuntimeService ينفذ نص CustomSql مباشرة، لكن الفحص الحالي لم يجد فيه enforcement يثبت SELECT-only أو parameter policy أو timeout. لذلك وجوده لا يساوي Query Runtime معتمدًا وآمنًا.
حد القرارالاختيار ليس بين «وجود مكان» و«عدم وجوده»، بل بين اعتماد نص SQL خام وتوحيد ربطه، أو ترقية الخاصية إلى Query Definition مهيكلة/متحققة تحافظ على semantics وتسمح بالحوكمة.
الحكم المؤقتفجوة تحويل مؤكدة، وأثرها المعماري لم يُحسم. لا نقرر بعد نقل SQL الخام؛ قد يكون المطلوب تحويل semantics الاستعلام إلى Query/View metadata أو مصدر بيانات داخلي آمن.
