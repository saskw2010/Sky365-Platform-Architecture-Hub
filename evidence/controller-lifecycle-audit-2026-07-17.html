تدقيق دورة حياة الـController الحالية

فحص مصدر بتاريخ 2026-07-17 — تخزين، تعديل، إصدار، Snapshot وتحميل Runtime.

النتيجة

نموذج Database-first موجود فعليًا، لكنه لم يُغلق الحلقة مع الـRuntime بعد. الـStudio يملك Controller Aggregate وجداول Versions داخل قاعدة البيانات، بينما VGem3 يحمّل DynamicScreenSchema من ملف JSON مباشرة. المطلوب تطوير جسر Publish/Export بين المسارين، لا استبدال الـController الحالي.

الموجود حاليًا Studio → System.SysControllers + Fields + Versions Runtime VGem3 → resolve schema path → JSON file → DynamicScreenSchema ↘ Code On Time XML fallback الحلقة المستهدفة DB Controller version → validate/publish/export → JSON schema → Runtime

1. التخزين في قاعدة البيانات — Verified

  • SysController مربوط بجدول System.SysControllers ويحمل المصدر وUILayoutJson وActionsJson وQueryJson وCustomSql ورقم النسخة.
  • SysControllerField مربوط بجدول System.SysControllerFields.
  • SysControllerVersion مربوط بجدول System.SysControllerVersions ويحفظ ControllerSnapshotJson ورقم النسخة والملاحظات والمستخدم والتاريخ.
  • DesignerDbContext يعرّف DbSets للجداول الثلاثة.

الدليل: Sky365Offers.Domain/Models/Studio/SysControllerEntities.cs:13-142 وSky365Offers.Data/DesignerDbContext.cs:23-25.

2. التعديل والإصدارات — Verified مع فجوات

الخدمة المسجلة في DI هي Business.Services.SysControllerConfigService. الإنشاء يبدأ Version 1 ويصنع Snapshot أوليًا. التعديل يحفظ الـController والحقول، لكنه يرفع النسخة تلقائيًا فقط عند تغير UILayoutJson أو ActionsJson.

التغييرحفظ DBVersion/Snapshot في الخدمة المسجلة
UILayout / Actionsنعمنعم
Fieldsنعملا يرفع النسخة وحده
QueryJson / CustomSqlالكيان يحتويهماUpdateController لا يحدّثهما
Title / Source / Description / Activeنعملا Snapshot تلقائيًا إلا مع Layout/Actions

الدليل: Sky365Offers.Business/Services/SysControllerConfigService.cs:62-218 وتسجيلها في Sky365Offers.BlazorWeb/Program.cs:306.

3. يوجد مسار Studio ثانٍ — Partial / خطر ازدواج

يوجد class بنفس الاسم داخل Sky365Offers.BlazorWeb.Features.Studio.Services. هذه النسخة تحفظ Layout وActions وEasyQuery منفصلة وترفع Version وتصنع Snapshot. البحث لم يجد تسجيلها في DI أو استخدامها من صفحات Studio التي تحقن Interface الخاص بخدمة Business.

المعنى: نية حفظ Query وإصداره موجودة، لكن هناك خدمتان بسلوك مختلف. يجب توحيدهما أو إثبات استخدام كل واحدة قبل اعتبار الـversioning مكتملًا.

4. Snapshot داخل DB وRollback — Verified

CreateVersionSnapshotAsync يقرأ الـController مع الحقول ثم يخزن JSON كاملًا داخل SysControllerVersions. RevertToVersionAsync يفك الـJSON ويعيده عبر UpdateController. هذا Snapshot تاريخي داخل قاعدة البيانات، وليس ملف Runtime منشورًا.

فجوة: لا تظهر حالات Draft/Approved/Published، ولا hash/signature، ولا optimistic concurrency token في الكيانات المفحوصة.

5. تحميل الـRuntime — Verified ومنفصل

Sky365DynamicRuntimeService.LoadSchemaAsync يحل مسار schema ثم يقرأ ملف JSON إلى DynamicScreenSchema. إذا لم يجد الملف يحاول Code On Time XML fallback. VGem3 وDynamicSchemaView يستدعيان هذه الخدمة.

الدليل: Sky365Offers.BlazorWeb/Features/DynamicEngine/Services/Sky365DynamicRuntimeService.cs:568-605 وRuntimeVGem3/Pages/DynamicRuntimeVGem3.razor.cs:249.

لم نجد: Publisher يحوّل SysController المعتمد إلى JSON الذي يقرأه الـRuntime، أو Runtime يختار نسخة Published من DB.

القرار التطويري

  1. الإبقاء على SysController في قاعدة البيانات كمصدر الحقيقة.
  2. توحيد خدمتي Controller Config في عقد واحد وسلوك version واحد.
  3. اعتبار كل تغيير مؤثر في Runtime سببًا لإصدار: fields، query، SQL، layout، actions، source والسياسات.
  4. إضافة دورة حالة واضحة للنسخة بعد اعتماد تفاصيلها.
  5. إنشاء Publisher يحوّل نسخة DB المحددة إلى DynamicScreenSchema متوافق مع الـRuntime الحالي.
  6. ربط الملف المنشور بـControllerId وVersion وhash ثم ضبط cache invalidation والrollback.

توثيق فقط؛ لم يتغير Production code.

السؤال التالي — بند واحد

هل كل Save داخل Builder يصبح نسخة منشورة فورًا، أم نفرّق بين Draft revisions كثيرة وPublished version واحدة لا يصل إليها الـRuntime إلا بعد اعتمادها؟
