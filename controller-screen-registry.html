Discovery lifecycle

root system registry -> system module registry -> module controllers.catalog.json -> DynamicControllerCatalogService -> normalize + deduplicate by moduleKey/schemaName -> unified controllers.catalog.json |-- screen runtime URL |-- dynamic rows API `-- schema preview URL

The unified catalog is generated output, not the authoring source. Module ownership must remain intact because dropping moduleKey can select the wrong schema.

Findings and gaps

  • Catalog coverage gap: the authored schema footprint is much larger than the runtime discovery catalog. Each excluded file must be classified as registry/governance, duplicate, non-screen metadata, intentionally hidden, invalid or missing onboarding.
  • Code On Time coverage gap: only a subset of Code On Time-derived JSON is represented in the unified catalog.
  • System identity drift: SystemGroup sometimes contains a technical key and sometimes a display label. Grouping by this field therefore fragments one logical system.
  • Runtime evidence gap: generated URLs show intended reachability but do not prove successful rendering, CRUD or authorization.

Future roadmap

Build a generated evidence registry keyed by canonical systemKey/moduleKey/schemaKey, classify every schema file, test one golden screen per module, and record runtime/API/preview results without mutating application data.
