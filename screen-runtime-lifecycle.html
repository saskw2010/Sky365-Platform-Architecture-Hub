Controller-to-Screen Runtime Lifecycle

Purpose: trace one Controller from discovery to human UI, CRUD API, workflow and agent tooling.

End-to-end sequence

1. Root system registry -> system module registry 2. Module catalog -> unified Controller catalog 3. User/agent chooses Controller -> schemaName + moduleKey + systemGroup + optional profile 4. Runtime resolves schema path -> JSON DynamicScreenSchema or Code On Time fallback conversion 5. Profile overlays base schema 6. Authorization and lookup contract checks 7. Mode dispatch |-- list: BuildRows + search + workflow row status |-- view/edit: GetById + lookups + detail schemas `-- create: CreateEmptyModel + defaults + lookups 8. UI renderer builds fields/sections/actions/details 9. Save -> UI rules -> pre-save rules -> CRUD/persistence -> post-save actions 10. Linked surfaces |-- workflow command/review |-- report builder/viewer |-- dynamic REST/OpenAPI/Swagger `-- agent skills and Controller tools

Stage 1 - Discovery

Input: root systems, module registries and module-owned Controller catalogs.

Runtime: DynamicControllerCatalogService, portal catalog providers and runtime index providers.

Output: normalized catalog item with schema, module, system/display context and generated URLs.

Failure risks: missing onboarding, catalog coverage gaps, duplicate scoped identities and display label/key drift.

Stage 2 - Route and scope

The primary VGem3 screen routes are:

  • /sky365-dynamic/runtime/vgem3/schema/{SchemaName}
  • /sky365-dynamic/runtime/vgem3/schema/{SchemaName}/{Mode}
  • /sky365-dynamic/runtime/vgem3/schema/{SchemaName}/{Mode}/{Id}

Catalog routes add system/module context. The runtime may also receive a profile key and workflow return context. Scope must survive every generated link; losing moduleKey or systemGroup can select another schema with the same name.

Stage 3 - Schema resolution

Sky365DynamicRuntimeService.LoadSchemaAsync resolves the scoped path, deserializes DynamicScreenSchema, or can map a Code On Time Controller through its fallback parser. A requested DynamicSchemaProfile then overlays list/form fields, defaults, details and field behavior.

Lookup definitions are validated. Explicit Primary/Fallback contracts require endpoint identity; legacy source-only lookups are accepted with warnings.

Stage 5 - Dynamic rendering

VGem3 and reusable Dynamic Engine components consume:

  • field type and labels to select controls;
  • sections/layout metadata to compose the form;
  • visibility/read-only expressions and dependencies;
  • lookup metadata, including cascading filters and quick add;
  • detail schemas for master-detail blocks;
  • view/actions/profile metadata for mode-specific experience.

Some business-specific checks still appear inside the VGem3 page code. These are deviations from a purely metadata-compiled runtime and should be cataloged before claiming full genericity.

Stage 6 - Save lifecycle

User changes field -> dependency/lookup refresh -> UI actions and conditional state -> optional local buffered draft -> SaveAsync -> business-flow validation -> pre-save validation/mutation rules -> CreateAsync or UpdateAsync -> master/detail persistence -> many-to-many persistence -> post-save actions -> return to workflow or list

Create/update call the runtime with effective schema, module and system context from the screen. The current HTTP Dynamic API does not preserve the same full scope; see ADR-0004.

Stage 7 - Dynamic REST API

The /api/dynamic group requires AdminAccess and exposes catalog, navigation graph, dataset comparison/projections, schema description, CRUD rows, OpenAPI and Swagger.

Stage 8 - Workflow, reports and agents

  • Workflow: WorkflowCode enables row status icons, workflow navigation and edit gates.
  • Reports: schema identity opens report builder/viewer and selects metadata-defined templates.
  • Agents: skills and Dynamic Tool Gateway discover the same catalog, describe fields, search records and open screens/records.
  • Semantic graph: Controller, schema, fields and linked artifacts become impact/query relationships.

Roadmap

Make scope immutable across routes, API, tools and caches; compile schemas before execution; move business-specific VGem3 branches into metadata/extensions; add non-mutating golden screen probes.
