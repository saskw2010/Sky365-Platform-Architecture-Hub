ADR-0006: Define Canonical Security Policy Inheritance Before Engine Promotion

Proposed — 2026-07-17

Context

Security is distributed across ASP.NET policies, system/module/catalog roles, actions, workflows, dashboards, reports and tools. The graph records requirements but does not enforce them. Dynamic API uses broad AdminAccess, while module role evaluation permits access when roles are absent and after caught lookup errors.

Decision

Before promoting Security/Permission to an Engine, define one versioned inheritance contract across system, module, Controller, profile, view/action, field/row and tool/integration operation. Every runtime returns an explicit decision and reason. Protected-scope resolution errors require an intentional, testable failure policy.

Required contract

  • Immutable resource/action identity and tenant scope.
  • Precedence/merge rules for roles, policies, denies and overrides.
  • Separate read/create/update/delete/execute/export/admin decisions.
  • Adapters for UI, API, data, workflow, dashboard, report, tool and integration.
  • Decision traces and golden allow/deny/error cases.

Consequences

Graph edges remain descriptive until enforcement consumes them. Existing route policies and role checks are evidence, not proof of a consolidated Engine. This ADR changes no runtime behavior.

Alternatives rejected

Graph representation as enforcement; UI hiding as security; or one global Admin policy as the sole MDAP authorization model.

Review trigger

Review when a canonical policy model and at least two cross-surface adapters are verified. See Partial Contracts.
