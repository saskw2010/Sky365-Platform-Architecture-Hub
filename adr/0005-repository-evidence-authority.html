ADR-0005: Assign Explicit Evidence Authority to SKY365 Repositories

Accepted - 2026-07-17

Context

Repository names and historical generated code can be mistaken for current platform ownership. The inspected repositories differ sharply in age, purpose and executable content.

Decision

  • Sky365Offers: primary current executable evidence until superseded by an explicitly governed platform repository.
  • sky365ERP: legacy Code On Time behavior and migration evidence; never automatic current-state proof.
  • sky365API: no implementation authority at the inspected README-only revision.
  • Sky365-knowledge-hub: authoritative for vision, vocabulary, research and roadmap intent; implementation status requires source/runtime corroboration.
  • Sky365-Platform-Architecture-Hub: maintained reconciliation of evidence, findings, decisions and gaps.

Consequences

  • Every claim records repository and revision.
  • Legacy features are labeled Legacy Evidence until a current counterpart is verified.
  • Empty or named repositories do not imply an implemented engine.
  • Cross-repository Controller migration requires structural evidence.
  • Sensitive historical configuration is referenced as a risk without copying secrets.

Alternatives rejected

Newest documentation wins: rejected because plans are not runtime evidence. Largest repository wins: rejected because generated legacy volume and duplicate artifacts distort architectural authority.

Review trigger

Review when a dedicated platform/runtime repository becomes canonical, when revisions materially change, or when an official repository ownership policy is adopted. See Repository Lineage.
