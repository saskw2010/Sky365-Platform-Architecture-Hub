ADR-0003: Make Controller Conversion Fidelity Explicit

Proposed - 2026-07-17

Context

Three Code On Time conversion paths currently map different subsets of the same XML contract. Silent loss is possible for views, actions, rules, lookup details, permissions and workflow metadata.

Proposed decision

Every Controller conversion must produce a versioned conversion report alongside normalized output. The report must identify source revision, compiler version, preserved constructs, defaults/guesses, unsupported constructs, warnings and errors. No conversion should be described as successful without a declared fidelity level.

Fidelity levels

  • F0 Rejected: source cannot be safely identified or parsed.
  • F1 Structural: identity and fields only; generic inspection, not operational parity.
  • F2 CRUD: keys, types, nullability and lookups validated for CRUD.
  • F3 Experience: views, layout, actions and conditional behavior preserved.
  • F4 Operational: rules, permissions, workflows and integration hooks preserved and tested.
  • F5 Conformant: normalized output passes golden runtime/API/UI comparison for the supported source subset.

Consequences

  • Existing import paths remain usable, but their output becomes honestly classified.
  • Guessed fields such as Id/Name are surfaced as warnings.
  • The proposed Controller Intermediate Representation becomes the comparison target.
  • Studio can show users what will be lost before import.

Alternatives rejected

Treat any parsed XML as successful: rejected because syntactic success is not behavioral parity. Remove legacy imports immediately: rejected because Code On Time Controllers are valuable platform knowledge and active screen sources.

Status

Proposed pending a representative golden Controller suite and agreement on the supported Code On Time subset. See Conversion Fidelity Matrix and ADR-0002.
