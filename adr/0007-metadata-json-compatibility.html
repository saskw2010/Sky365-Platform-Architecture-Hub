ADR-0007: Version Metadata JSON Before Enforcing a Canonical Serialization Contract

Proposed — 2026-07-17

Context

The central loader reads properties case-insensitively and does not configure strict unknown-member rejection. Writers differ: some use default property naming while designer paths use camelCase. Dashboard layout supports legacy and current shapes.

Decision

Treat existing JSON as compatibility-oriented authored metadata. Do not retroactively enforce one strict casing or unknown-property rule. Introduce a versioned canonical contract through an explicit migration/compiler boundary with diagnostics and source maps.

Required future contract

  • Version discriminator and supported-version policy.
  • Canonical naming for newly compiled artifacts.
  • Defined defaults, null and unknown-property behavior.
  • Compatibility readers/migrations.
  • Source/property diagnostics and golden round-trip tests.

Consequences

Permissive loading is compatibility evidence, not proof of a normative JSON standard. A future strict compiler can emit canonical artifacts without silently breaking legacy/authored inputs. This ADR changes no runtime code.

Alternatives rejected

Immediately reject unknown fields; infer the standard from the newest writer; or treat C# classes as a versionless immutable JSON contract.

Review trigger

Review when CIR/compiler versioning and migration tests exist. See Metadata Property Reference.
