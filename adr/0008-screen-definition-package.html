ADR-0008: Adopt a modular Screen Definition Package

Proposed target architecture · owner direction · 2026-07-17

Context

SKY365 already renders metadata-defined fields, sections, views, actions and rules through generic Blazor components. Screen Studio is therefore an authoring surface for contracts, not a source-code generator. The current schema contains useful primitives, but screen composition, data-source definitions, permissions and runtime artifacts remain distributed.

Decision

Model a screen as one logical Screen Definition Package with modular authoring contracts and one compiled Runtime Snapshot. Keep Controller responsible for data/behavior and Screen responsible for composition/presentation. Make View a first-class versioned binding between a data source, presentation, actions and access policy.

  • A Screen may reference multiple Controllers.
  • A Controller may be reused by multiple Screens.
  • Field definition is canonical; presentation is View-specific.
  • Actions are the intersection of source capability, View configuration, user capability and runtime condition.
  • Read Model and Write Model are independent contracts.
  • Capabilities are the target authorization vocabulary; direct Roles remain a compatibility mechanism.
  • Authoring artifacts are modular; publish output is unified, validated and versioned.
  • AI produces typed plans through skills and validators, not ungoverned final JSON.

Status qualification

This is a target architecture decision, not evidence that the complete package/compiler exists. Existing evidence includes DynamicScreenSchema, DynamicViewSchema, sections/tabs, action/rule metadata, role fields, EasyQuery integration and separate dashboard/report definitions.

Consequences

The platform needs JSON Schemas, canonical identities, reference validation, capability inheritance, DataSourceDefinition/SkyQueryPlan, View presentation contracts, compiler diagnostics, migrations, publish/approval and conformance tests. EasyQuery, Wren or future designers are adapters; SKY365 owns the contracts and execution policy.

Alternatives rejected

One monolithic Controller: creates coupling and poor reuse/version control. Unrelated Controller and UI files: loses traceable ownership. Generate bespoke screen code: conflicts with the metadata runtime model. Let vendor JSON become canonical: creates lock-in.

Review triggers

Review after a Golden Screen Package, schema compiler and capability policy prototype exist, or if current runtime constraints prove that multi-Controller composition cannot be supported safely.
