ADR-0008: Keep the Controller as the database source of truth

Accepted direction · owner clarification · 2026-07-17

Context

SKY365 already renders metadata-defined fields, sections, views, actions and rules through generic Blazor components. Screen Studio and the specialized builders edit metadata; they do not generate bespoke screen code. Controller metadata is stored and versioned in the database, while JSON is exported for fast runtime consumption.

Decision

Continue evolving the existing model. The Controller is the aggregate and authoritative source of truth. It may contain the full screen definition: data source, fields, relations, views, layout, queries, actions, policies, workflow bindings and runtime metadata. These remain clear internal contract sections, but they are not independent authoritative files.

  • Builders load the working Controller version from the database and update their owned sections.
  • Every saved change creates or advances database-managed version history.
  • Publishing validates the aggregate and exports the latest approved version as a unified JSON Runtime Snapshot.
  • The Runtime reads the exported snapshot for speed and may cache it; it does not become the authoring authority.
  • The snapshot carries Controller identity, version, publication metadata and integrity information so it can be traced back to its database version.
  • View remains a first-class section inside the Controller, with query, presentation, actions and access rules.
  • Field definition is canonical inside the Controller; presentation is View-specific.
  • Read Model and Write Commands may be separated inside the same Controller aggregate.
  • AI and Studio use typed commands and validators against the same database model.

Lifecycle

Database Controller draft → Builder edit → validate → save new version → approve/publish → export JSON snapshot → cache/load → Runtime. Rollback selects an earlier database version and republishes a new traceable snapshot; editing an exported JSON file directly is not an authoritative change.

Status qualification

This decision preserves the current direction and refines its contracts. Existing evidence includes DynamicScreenSchema, DynamicViewSchema, sections/tabs, action/rule metadata, role fields, EasyQuery integration and runtime JSON loading. The exact database version, approval and snapshot integrity implementation still requires source-level verification.

Consequences

The platform needs explicit ownership rules so concurrent builders do not overwrite unrelated sections, optimistic concurrency, version status such as Draft/Approved/Published/Retired, schema migrations, validation diagnostics, deterministic snapshot export, cache invalidation and rollback tests. EasyQuery, Wren or future designers are adapters; SKY365 owns the Controller aggregate and execution policy.

Alternatives rejected

Independent authoring files as sources of truth: conflicts with database-managed Controller authoring and versioning. JSON as the editable authority: risks divergence from the database. Generate bespoke screen code: conflicts with the metadata runtime model. Let vendor JSON become canonical: creates lock-in.

Review triggers

Review after the database version lifecycle and export path are fully traced from source, and after one Golden Controller is followed from Builder edit through publication, snapshot loading and rollback.
