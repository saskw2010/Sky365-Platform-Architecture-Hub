ADR-0004: Preserve Full Controller Scope Across Every Runtime Surface

Proposed - 2026-07-17

Context

VGem3 calls the runtime with schema, module and system context. The Dynamic REST CRUD endpoints accept schema and module only, while the OpenAPI endpoint also accepts system group. Catalog data itself mixes technical and display values in SystemGroup.

Risk

When schema names repeat across systems, UI, API, tools, reports and workflows can resolve different definitions. Authorization and cache isolation can also become inconsistent when scope is optional or presentation labels act as identity.

Proposed decision

Represent runtime identity as an immutable scoped value containing systemKey, moduleKey, schemaKey and optional profileKey. Every route, API, tool, workflow/report link, cache key, trace and graph node must either carry this value or resolve it once from a globally unique Controller ID.

Rules

  • Display titles and localized SystemGroup labels are never identifiers.
  • Scope may not silently fall back across unrelated systems.
  • Generated URLs include enough identity to reproduce the same schema resolution.
  • Authorization evaluates the same scoped operation used for data resolution.
  • OpenAPI contracts and tool definitions record the scope they describe.
  • Traces record both canonical keys and display labels.

Consequences

  • Some existing URLs require compatibility redirects or resolution adapters.
  • Catalog normalization and data cleanup are prerequisites.
  • Cross-system duplicate schema names become safe and observable.
  • UI/API/tool parity becomes testable.

Status

Proposed pending a duplicate identity census and compatibility impact report. Related: Screen Lifecycle, ADR-0002.
