Core Runtime Engines

Reference for engine boundaries directly verified in executable source.

Relationship map

Controller Catalog + JSON metadata -> Dynamic Runtime |-- UI Engine |-- CRUD + Persistence |-- Lookup + Validation + Business Rules |-- API discovery |-- Workflow links |-- Dashboard / BI |-- Report Engine `-- Integration Engine Metadata Graph observes and links all metadata domains.

1. Metadata and Controller Catalog Engine

Purpose: create the unified discovery surface for module-owned Controllers.

Responsibilities: read module catalogs and modular Controller files, normalize values, deduplicate, enrich URLs and optionally persist the master catalog.

Inputs: root/system registries, module catalogs and Controller metadata. Outputs: DynamicControllerCatalogSnapshot, runtime/API/preview URLs.

Runtime: DynamicControllerCatalogService.BuildSnapshotAsync. Metadata: system, module, schema, node type, roles, visibility and navigation attributes.

Dependencies: IDynamicStorageProvider, JSON serialization and filesystem. Extension points: module-local catalogs and normalization rules.

Example: a module Controller becomes a catalog Screen with a module-aware runtime URL. Roadmap: canonical identity validation, coverage report and provenance hashes.

2. Dynamic Controller Runtime

Purpose: materialize metadata into an executable application surface.

Responsibilities: resolve scoped schema paths and profiles; load schemas; query rows; create empty models; execute CRUD; resolve lookups; check roles/modules; run rules/actions.

Inputs: schema name, module key, system group, profile, user and record values. Outputs: DynamicScreenSchema, rows, record models, lookup options and operation results.

Runtime: Sky365DynamicRuntimeService. Metadata: fields, views, sections, details, lookups, rules, actions, roles and connection/table identity.

Dependencies: catalog/storage, Code On Time parser, SQL, configuration and authorization context. Extension points: schemas, profiles, lookup sources and actions.

Example: LoadSchemaAsync followed by BuildRowsAsync drives a list screen. Roadmap: compile to a versioned CIR and separate query/execution policies.

3. Dynamic UI Engine

Purpose: render forms, grids, detail blocks and navigation from schema metadata.

Responsibilities: choose field controls, compose list/editor/detail views, expose search/lookups, execute UI actions and present validation state.

Inputs: DynamicScreenSchema, record values, route/profile and user context. Outputs: Blazor UI and user commands.

Runtime: DynamicFieldRenderer, DynamicSchemaView, NoorAutoCrud and runtime layouts. Metadata: field type/label/order, views, sections, actions, visibility and lookup definitions.

Dependencies: Dynamic Runtime, component library and route resolver. Extension points: new renderer types, view profiles, custom components and layout metadata.

Roadmap: renderer conformance tests and an explicit metadata-to-component registry.

4. CRUD and Persistence Engine

Purpose: execute metadata-scoped data operations, including master/detail persistence.

Responsibilities: list/get/create/update/delete; atomic, immediate and hybrid save modes; detail batches; many-to-many persistence; column/key resolution.

Inputs: schema, master/detail values, record ID and persistence mode. Outputs: rows, IDs, success state or PersistenceResult.

Runtime: CRUD methods in Sky365DynamicRuntimeService and DynamicPersistenceEngine. Metadata: table/schema, key, identity, fields, details, many-to-many and connection reference.

Dependencies: SQL connection/transaction and schema loader. Extension points: persistence strategies and storage providers.

Roadmap: command model, optimistic concurrency contract, policy hooks and database-independent provider boundary.

5. Business Rule and Validation Engine

Purpose: apply declarative behavior before, during and after data interaction.

Responsibilities: evaluate pre-save rules, field/list filters, expressions, UI actions and post-save actions.

Inputs: schema rules, trigger, changed field and row context. Outputs: validation outcomes, transformed behavior and post-save effects.

Runtime: DynamicBusinessRuleEngine, DynamicExpressionEvaluator and runtime rule/action methods. Dependencies: schema and runtime context.

Extension points: trigger types, expression operators and action executors. Roadmap: typed rule AST, determinism classification and rule trace output.

6. Workflow Engine

Purpose: manage record workflows, steps, assignments, actions and state transitions.

Responsibilities: start workflows; submit actions; transition/activate/finalize steps; expose status/history/inbox/definitions; cancel; update entity status; dispatch notifications.

Inputs: workflow definition/code, record identity/data, user, roles and action. Outputs: instances, step state, pending work, history, entity status and notification events.

Runtime: WorkflowEngineService and workflow UI components. Metadata: definitions, steps, destinations, assignments, roles and schema links.

Dependencies: SQL, notification dispatcher, metadata files and runtime record loader. Extension points: transition destinations, assignment rules and dispatchers.

Roadmap: compile workflow metadata, add version migration rules and end-to-end replay evidence.

7. Dashboard and BI Engine

Purpose: resolve dashboard metadata into data-backed analytical widgets.

Responsibilities: load dashboard definitions, resolve data sources/date ranges, execute approved stored procedures and render chart/pivot/widget components.

Inputs: dashboard/widget definition, filters, date range and data-source identity. Outputs: widget datasets and dashboard views.

Runtime: DynamicDashboardService, DashboardDataSourceResolver, stored-procedure catalog/executor and renderers. Dependencies: dashboard registry, SQL/API sources and component renderers.

Extension points: widget types and governed data-source resolvers. Roadmap: semantic measures/dimensions and query-cost/security policy.

8. Report Engine

Purpose: create and run metadata-defined record and tabular reports.

Responsibilities: templates, component library, datasets, filters, builder/viewer and report assets linked to Controllers.

Inputs: Controller/schema, report template, fields and filters. Outputs: report dataset and rendered report.

Runtime: DynamicReportTemplateService, DynamicReportDataSetFactory, DynamicReportFilterEngine and report pages. Extension points: components, templates and output adapters.

Roadmap: report compiler, versioned contracts, authorization inheritance and export evidence.

9. Integration Engine

Purpose: isolate declarative external-system actions from core UI and persistence code.

Responsibilities: resolve and execute integration definitions with runtime context.

Inputs: integration metadata, record/context and credentials policy reference. Outputs: external call result or controlled failure.

Runtime: DynamicIntegrationEngine. Dependencies: HTTP/external adapters, configuration, security and audit.

Extension points: provider adapters and integration action types. Roadmap: connector registry, retry/idempotency, secrets isolation and observability.

10. Metadata Graph Engine

Purpose: provide a connected architectural view of metadata rather than isolated JSON documents.

Responsibilities: ingest systems, modules, Controllers, schemas, fields, workflows, navigation, dashboards and reports; link lookups and cross-domain relationships; expose related-node queries.

Inputs: dynamic metadata tree. Outputs: MetadataGraphSnapshot, typed nodes/edges and indexes.

Runtime: MetadataGraphService. Dependencies: catalog/schema models and JSON filesystem.

Extension points: node/edge kinds and new metadata ingestors. Roadmap: provenance, semantic vocabulary, impact analysis, graph validation and controlled vector projection.

Cross-engine gaps

  • Canonical Controller identity and compilation are not yet uniform.
  • Authorization exists but inheritance across UI, CRUD, reports, dashboards, tools and integrations needs one policy contract.
  • Generated links and source presence are weaker than runtime verification.
  • Observability, provenance, version compatibility and failure semantics are inconsistent across engine boundaries.
