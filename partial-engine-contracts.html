API, Search, Security, Semantic and Compiler Contracts

Normalized evidence cards for remaining partial or distributed platform boundaries.

Relationship map

Controller XML/JSON → distributed conversion/loading → runtime schema → CRUD API + SQL filtering + UI → metadata graph → tools/agents; authorization enters through route, module and metadata-specific seams

1. Dynamic API Engine

Status: advanced partial runtime; compiler/governance incomplete.

Purpose
Expose catalog, schema, CRUD rows and generated OpenAPI/Swagger.
Responsibilities
Catalog/navigation datasets, schema description, list/get/create/update/delete and OpenAPI generation from fields.
Inputs → outputs
schemaName, optional moduleKey, search/body/id → JSON results; OpenAPI additionally accepts systemGroup.
Runtime
/api/dynamic minimal API group backed by Sky365DynamicRuntimeService and DynamicRuntimeApiDescriptionService.
Metadata used
Screen/table/id, fields, type, required/read-only/max-length/placeholder and catalog routes.
Dependencies
Catalog, schema loader, CRUD/persistence, OpenAPI library and authorization.
Extension points
Runtime/description interfaces and catalog services.
Failure semantics
404 for missing get/update/delete; no complete versioned error model.
Gaps
CRUD omits systemGroup although OpenAPI accepts it; integer-only route IDs; Swagger URL omits systemGroup; no versioning, idempotency, typed action contracts or Controller policy compilation.

Controller relation: fields compile into object/write schemas and CRUD behavior, but identity and policy are not canonical Controller contracts.

2. Search Engine

Status: runtime filtering; independent Search Engine not established.

Purpose
Filter Controller rows using user text and declarative list filters.
Responsibilities
Select physical list columns, apply filters/free text and order by ID.
Inputs → outputs
Resolved screen schema + search + ListFilters → row dictionaries.
Runtime
BuildRowsAsync/BuildRowsGenericAsync; Centers/Complexes have special domain-service paths.
Metadata used
Fields.Name, InList, IdField, ListFilters(Field, Operator, Value).
Dependencies
Schema loader, DB column inspection, SQL and parameterized execution.
Extension points
Domain special cases; no search-provider adapter.
Failure semantics
Throws when no schema fields match DB columns; identifier normalization rejects unsafe names; unsupported list operators fall back to equality.
Gaps
No index, analyzers, ranking, pagination, facets, typo handling, incremental updates, ACL plan or relevance tests. Free text uses parameterized LIKE over TRY_CONVERT(nvarchar(max), column).

Controller relation: fields indirectly select searchable columns; no explicit weights/analyzers/search policy exists.

3. Security and Permission Engine

Status: distributed controls; consolidated Engine not established.

Purpose
Authorize systems, modules, Controllers, actions, workflows, dashboards, reports, APIs and tools.
Responsibilities
Named policies, route gates, role checks, module role lookup and metadata role/permission graph ingestion.
Inputs → outputs
Principal + policy/resource/action/scope + roles → allow/deny, route challenge/forbidden or graph requirements.
Runtime
Dynamic API requires AdminAccess; MCP has separate policies; role helpers use claims and module registry.
Metadata used
System/module/catalog/action/workflow/dashboard/report role and policy hints.
Dependencies
ASP.NET Identity/authorization, registry topology, current principal and every consuming runtime.
Extension points
Named policies and local metadata checks; no single evaluator.
Failure semantics
Empty role strings permit access. Module authorization permits access when roles are absent and returns allow after caught lookup errors. AdminAccess protects Dynamic API, but does not prove per-Controller enforcement elsewhere.
Gaps
No inheritance/precedence matrix across system → module → Controller → view/action → field/row; no uniform tenant scope/deny reason; graph edges describe but do not enforce.

See ADR-0006.

4. Metadata Graph and Semantic Engine

Status: verified metadata graph; Semantic Engine foundation only.

Purpose
Connect systems, modules, Controllers, schemas, fields, lookups, rules/actions, workflows, navigation, dashboards, reports, roles and permissions.
Responsibilities
Build/warm/rebuild cached snapshot, typed nodes/edges/indexes and related-node queries.
Inputs → outputs
Registries and linked JSON models → MetadataGraphSnapshot, statistics and typed query results.
Runtime
MetadataGraphService plus hosted warm-up; in-memory snapshot.
Metadata used
Linked identities, relationships and role/permission tokens.
Dependencies
Catalog/schema/workflow/dashboard/report loaders, caching and logging.
Extension points
IMetadataGraphService and typed node model.
Failure semantics
Unknown IDs return null/empty; no durable graph transaction or lineage/error ledger.
Semantic gaps
No governed vocabulary, synonyms, multilingual concepts, entity resolution, confidence, ownership, versioned mappings, ontology constraints or semantic query language. Permission nodes are descriptive.

Controller relation: Controller/field/rule identities are graph anchors, making this the strongest Semantic precursor.

5. Controller / Metadata Compiler and Builders

Status: distributed compiler-like pipeline; owned compiler not established.

Purpose
Transform Code On Time XML, DB metadata or authored JSON into validated canonical Controller runtime artifacts.
Responsibilities
Parsing/mapping, onboarding, batch/DB import, schema loading, scaffolding, catalog normalization and routes.
Inputs → outputs
XML/DB/POCO/JSON + scope → different JSON, DB, XML, catalog and route artifacts depending on path.
Runtime
CodeOnTimeXmlParser, onboarding/import services, SysControllerConfigService, scaffolding, schema/catalog/route services.
Metadata used
Controller, table, fields, lookup hints and—only in richer paths—views/categories/actions.
Dependencies
Filesystem, serializers, DB config persistence, catalog and runtime.
Extension points
Parser/import/service interfaces; no compiler phase/plugin model.
Failure semantics
Path-specific diagnostics/fidelity; no common validation report, source map or loss warning.
Gaps
No CIR, phase contract, compatibility model, deterministic output, golden suite or single owner. Route/report Builders are not Controller compilers.

Cross-engine example

A dynamic rows request passes the AdminAccess route gate, loads the Controller schema, derives searchable physical columns, runs parameterized SQL filtering and returns dynamic dictionaries. OpenAPI separately derives field schemas. The graph can describe roles but does not automatically enforce those edges on CRUD. Thus API, Search, Security and Semantic remain related partial boundaries.

Roadmap

  1. Use scoped Controller identity across CRUD/OpenAPI.
  2. Define API version/error/idempotency/action contracts.
  3. Add explicit search fields, weights, policies and provider interface.
  4. Define and test security inheritance across every surface.
  5. Layer governed semantic vocabulary/mappings on the graph.
  6. Create CIR/compiler phases and golden Code On Time parity tests.
