Screen Definition Package — Target Architecture

عقد الشاشة الذي يفهمه Screen Studio والـAI والـRuntime دون توليد Screen Component مخصص.

القرار

Screen Studio محرر عقود JSON وليس مولّد كود. الشاشة كيان منطقي واحد، لكن تعريفها أثناء التأليف مقسّم إلى عقود مستقلة قابلة لإعادة الاستخدام والتحقق والإصدار. عند النشر تُجمع في Runtime Snapshot واحد.

Screen Definition Package ├── Screen Manifest ├── Controller References + Field Catalog ├── Query / Data Source Definitions ├── First-class View Definitions ├── Layout + UI/UX Rules ├── Actions + Access Policies ├── Workflow / Events └── Runtime Metadata ↓ compile + validate screen.runtime.json ↓ Blazor Runtime: Grid / Form / Chart / Tabs / Query Builder

حدود المسؤولية

العقدالسؤال الذي يجيب عنهالمحتوى
Controllerما البيانات وما السلوك الممكن؟Entity/data source, fields, relations/lookups, queries, validation, business rules, workflow bindings, capabilities وWriteMap.
Screenكيف نركب تجربة المستخدم؟Controllers aliases, tabs, views, default view rules, layout وnavigation.
Viewكيف نعرض Dataset محددة لمستخدم محدد؟type, controllerRef, queryRef, field presentation, actions, access, conditions وbindings.
Policyمن يستطيع رؤية أو تنفيذ ماذا؟Capabilities ومتطلبات Screen/Tab/View/Field/Action/Record/Query.
Studioكيف نؤلف العقود؟Grid/Form/Query/Visualization/Dashboard/Action/Access/Workflow builders.
Runtimeكيف ننفذ العقد المنشور؟تحميل Snapshot، policy evaluation، component resolution، query/persistence/action execution.

العلاقة الأساسية: Controller واحد يمكن أن يظهر في شاشات متعددة، وشاشة واحدة يمكن أن تجمع عدة Controllers تحت Screen Manifest واحد.

View كيان First-class

الـView ليست اسمًا مرادفًا لـGrid أو Form؛ هي binding بين Dataset وطريقة عرض وسياسة وصول وأفعال.

{
  "viewKey": "orders-grid",
  "type": "grid",
  "controllerRef": "orders",
  "dataSourceRef": "active-orders",
  "presentationRef": "views/orders-grid",
  "requires": ["sales.orders.read"],
  "actions": ["view", "edit", "new", "export"]
}

Field Catalog يعرّف معنى الحقل ونوعه ومصدره مرة واحدة، بينما كل View تملك presentation خاصة: Grid width/sort/filter، Form editor/layout/required، وChart role/aggregation.

الصلاحيات والأفعال

Role → grants Capabilities Screen / Tab / View / Field / Action / Query → requires Capabilities Runtime → evaluates policy + record predicate

أسماء Roles يمكن دعمها للتوافق، لكن العقد المستهدف يعتمد Capabilities. القرار النهائي لظهور الفعل:

VisibleAction = Source Capability AND View Action Enabled AND User Capability AND Runtime Condition

Studio يصدر Validation Warning إذا طلبت View فعلًا لا يسمح به المصدر. Database View وRaw Query افتراضيًا read-only، لكن يمكن تعريف Write Commands مستقلة؛ لذلك Mutability لا تُستنتج من نوع Read Source وحده.

Modular authoring، Unified runtime

وقت التأليفوقت التشغيل
screen.json، controller.json، fields.json، views.json، queries.json، actions.json، access.json.screen.runtime.json واحد، resolved وvalidated وversioned، مع dependency hashes وdiagnostics.
أفضل للـdiff وإعادة الاستخدام وتحرير Builder محدد وتقليل conflicts.أفضل للتحميل والـcache والتوقيع وعدم تنفيذ resolution مكلف عند كل فتح.

AI Screen Planning

User requirement → AI Screen Planner → typed plan (not free-form final JSON) → Screen Skills → schema + semantic + policy validation → modular JSON package → Studio preview + diagnostics → human approval → compiled Runtime Snapshot

Skills المستهدفة: Schema Discovery، Semantic Mapping، Screen Planning، Query Design، Form/Grid Design، Visualization، UX Rules، Access Policy، Mutability، Action Design، Validation وMigration. نفس العقود تستخدم من AI وCodex وStudio وCI validation.

EasyQuery وSemantic Query Gateway

Query Builder محرر لجزء Data Source/View وليس مالك الشاشة. الاتجاه:

Wren / Agent ↔ SkyQueryPlan ↔ EasyQuery Adapter ↔ EasyQuery Query JSON ↓ SKY365 validation + policy + compiler ↓ Grid / Report / Dashboard

SQL output مشتق. Read Query لا يحدد WriteMap. EasyQuery يمكن استبداله دون تغيير عقد الشاشة، لأن SKY365 تملك SkyQueryPlan وDataSourceDefinition.

خارطة التنفيذ بعد اعتماد Goal 1

  1. تثبيت terminology والهوية والإصدارات ومراجع Controller/View/DataSource.
  2. كتابة JSON Schemas للعقود، مع compatibility/migration policy.
  3. إنشاء Golden Screen Package يضم Controllerين وGrid/Form/Trend/Query وcapability policies.
  4. تعريف Compiler inputs/outputs/diagnostics وRuntime Snapshot، دون تغيير Runtime أولًا.
  5. بناء validators: references، mutability/actions، field/query policy، cycles وoutput schema.
  6. ربط EasyQuery عبر Adapter إلى SkyQueryPlan.
  7. توحيد Grid/Report/Dashboard على DataSourceRef وControlled Query Runtime.
  8. إضافة Component Registry وView-specific presentation ثم Screen Studio builders.
  9. إضافة AI typed planner/skills بعد استقرار العقود والـvalidators.
  10. تفعيل publish/approve/sign/cache/rollback ومجموعة conformance tests.

ليس ضمن هذه المرحلة: تنفيذ Production code، Fork لـNocoBase، React runtime pivot، أو تنفيذ SQL مولدة من AI مباشرة.

قرارات مفتوحة للنقاش واحدًا واحدًا

  1. هل Controller وField Catalog artifact واحد أم عقدان قابلان لإعادة الاستخدام؟
  2. هل View يمكن مشاركتها بين Screens أم presentation فقط هي القابلة للمشاركة؟
  3. ما minimum capability vocabulary للنسخة الأولى؟
  4. أين يعيش record-level predicate: Controller policy أم DataSource policy؟
  5. ما format Runtime Snapshot وكيف تُدار migrations والتوقيعات؟
