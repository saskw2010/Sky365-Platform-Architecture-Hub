Screen Definition Package — Target Architecture

عقد الشاشة الذي يفهمه Screen Studio والـAI والـRuntime دون توليد Screen Component مخصص.

القرار المصحح

الـController الموجود في قاعدة البيانات هو Aggregate ومصدر الحقيقة. Screen Studio والـBuilders لا تولّد كودًا ولا تجعل ملفات JSON مصدرًا بديلًا؛ كل Builder يقرأ نسخة الـController من قاعدة البيانات ويعدّل الجزء المسؤول عنه ثم يحفظ إصدارًا جديدًا. عند النشر تُصدّر آخر نسخة معتمدة إلى Runtime Snapshot واحد سريع.

Controller Aggregate — Database Source of Truth ├── Screen Manifest ├── Controller References + Field Catalog ├── Query / Data Source Definitions ├── First-class View Definitions ├── Layout + UI/UX Rules ├── Actions + Access Policies ├── Workflow / Events └── Runtime Metadata ↓ validate + version + approve + publish controller.runtime.json (derived snapshot) ↓ Blazor Runtime: Grid / Form / Chart / Tabs / Query Builder

حدود المسؤولية داخل الـController

هذه حدود منطقية تساعد الـBuilders والـRuntime والـAI، وليست ملفات مستقلة تمتلك الحقيقة خارج قاعدة البيانات.

القسمالسؤال الذي يجيب عنهالمحتوى
Controllerما البيانات وما السلوك الممكن؟Entity/data source, fields, relations/lookups, queries, validation, business rules, workflow bindings, capabilities وWriteMap.
Screenكيف نركب تجربة المستخدم؟Controllers aliases, tabs, views, default view rules, layout وnavigation.
Viewكيف نعرض Dataset محددة لمستخدم محدد؟type, controllerRef, queryRef, field presentation, actions, access, conditions وbindings.
Policyمن يستطيع رؤية أو تنفيذ ماذا؟Capabilities ومتطلبات Screen/Tab/View/Field/Action/Record/Query.
Studioكيف نؤلف العقود؟Grid/Form/Query/Visualization/Dashboard/Action/Access/Workflow builders.
Runtimeكيف ننفذ العقد المنشور؟تحميل Snapshot، policy evaluation، component resolution، query/persistence/action execution.

العلاقة الأساسية: الـController الواحد يمكن أن يحتوي تعريف الشاشة كاملًا. أي دعم مستقبلي لربط Controller آخر يكون Reference صريحًا يديره الـController الأساسي، وليس تحويل Screen Manifest منفصل إلى مصدر حقيقة جديد.

View كيان First-class

الـView ليست اسمًا مرادفًا لـGrid أو Form؛ هي binding بين Dataset وطريقة عرض وسياسة وصول وأفعال.

{
  "viewKey": "orders-grid",
  "type": "grid",
  "controllerRef": "orders",
  "dataSourceRef": "active-orders",
  "presentationRef": "views/orders-grid",
  "requires": ["sales.orders.read"],
  "actions": ["view", "edit", "new", "export"]
}

Field Catalog يعرّف معنى الحقل ونوعه ومصدره مرة واحدة، بينما كل View تملك presentation خاصة: Grid width/sort/filter، Form editor/layout/required، وChart role/aggregation.

الصلاحيات والأفعال

Role → grants Capabilities Screen / Tab / View / Field / Action / Query → requires Capabilities Runtime → evaluates policy + record predicate

أسماء Roles يمكن دعمها للتوافق، لكن العقد المستهدف يعتمد Capabilities. القرار النهائي لظهور الفعل:

VisibleAction = Source Capability AND View Action Enabled AND User Capability AND Runtime Condition

Studio يصدر Validation Warning إذا طلبت View فعلًا لا يسمح به المصدر. Database View وRaw Query افتراضيًا read-only، لكن يمكن تعريف Write Commands مستقلة؛ لذلك Mutability لا تُستنتج من نوع Read Source وحده.

Database authoring، Unified runtime

Controller Draft in Database → Builder loads owned section → edit + validate → save new database version → approve / publish → export controller.runtime.json → cache → Runtime
وقت التأليفوقت التشغيل
الـController Aggregate محفوظ في قاعدة البيانات. كل Builder يحرر Section محددًا مع optimistic concurrency، وتدير قاعدة البيانات التاريخ والحالة والإصدارات.controller.runtime.json مشتق واحد، resolved وvalidated وversioned، يحمل ControllerId وVersion وPublishedAt وintegrity hash.
قاعدة البيانات هي المرجع عند المراجعة والتعديل والrollback. تقسيم الأقسام منطقي داخل النموذج وليس تعددًا في مصادر الحقيقة.الملف والـcache لتحسين سرعة القراءة فقط. لا يُعدّل الملف مباشرة، ولا يتفوق على نسخة قاعدة البيانات.

AI Screen Planning

User requirement → AI Screen Planner → typed plan (not free-form final JSON) → Screen Skills → schema + semantic + policy validation → database Controller draft/version → Studio preview + diagnostics → human approval → published Runtime Snapshot

Skills المستهدفة: Schema Discovery، Semantic Mapping، Screen Planning، Query Design، Form/Grid Design، Visualization، UX Rules، Access Policy، Mutability، Action Design، Validation وMigration. نفس الأوامر المهيكلة والـvalidators تستخدم من AI وCodex وStudio، والحفظ النهائي يمر دائمًا عبر خدمة الـController في قاعدة البيانات.

EasyQuery وSemantic Query Gateway

Query Builder محرر لجزء Data Source/View وليس مالك الشاشة. الاتجاه:

Wren / Agent ↔ SkyQueryPlan ↔ EasyQuery Adapter ↔ EasyQuery Query JSON ↓ SKY365 validation + policy + compiler ↓ Grid / Report / Dashboard

SQL output مشتق. Read Query لا يحدد WriteMap. EasyQuery يمكن استبداله دون تغيير عقد الشاشة، لأن SKY365 تملك SkyQueryPlan وDataSourceDefinition.

خارطة التنفيذ بعد اعتماد Goal 1

  1. تثبيت terminology والهوية والإصدارات ومراجع Controller/View/DataSource.
  2. توثيق Database Controller schema ودورة Draft/Approved/Published/Retired وإثباتها من الكود.
  3. كتابة schemas لأقسام الـController، مع compatibility/migration policy.
  4. تتبع Golden Controller واحد من Builder إلى database version ثم JSON export ثم Runtime.
  5. تعريف Publisher inputs/outputs/diagnostics وRuntime Snapshot، دون تغيير Runtime أولًا.
  6. بناء validators: references، mutability/actions، field/query policy، cycles وoutput schema.
  7. ربط EasyQuery عبر Adapter إلى SkyQueryPlan.
  8. توحيد Grid/Report/Dashboard على DataSourceRef وControlled Query Runtime.
  9. إضافة Component Registry وView-specific presentation ثم Screen Studio builders.
  10. إضافة AI typed planner/skills بعد استقرار العقود والـvalidators.
  11. تفعيل publish/approve/sign/cache/rollback ومجموعة conformance tests.

ليس ضمن هذه المرحلة: تنفيذ Production code، Fork لـNocoBase، React runtime pivot، أو تنفيذ SQL مولدة من AI مباشرة.

قرارات مفتوحة للنقاش واحدًا واحدًا

  1. ما الجداول والخدمات الحالية التي تحفظ الـController ونسخه، وما حالة كل نسخة؟
  2. هل الحفظ الحالي ينشئ نسخة كاملة أم Delta، وكيف يمنع تعارض Builderين؟
  3. متى يصبح الإصدار Published، ومن يعتمد النشر؟
  4. ما الحقول الإلزامية لربط Snapshot بنسخة قاعدة البيانات والتحقق من سلامته؟
  5. كيف يحدث cache invalidation وrollback عند نشر نسخة جديدة أو سابقة؟
  6. ما minimum capability vocabulary للنسخة الأولى؟
  7. أين يعيش record-level predicate داخل Controller policy أم DataSource section؟
