Korzh EasyQuery + SonicGarden Blazer::Querygen

Purpose: determine which existing capability should become the SKY365 visual query foundation, what should be learned from AI-to-SQL projects, and whether SonicGarden/blazer-querygen warrants a production fork.

Executive judgment

Korzh EasyQuery is already the closest executable foundation inside SKY365. It provides a structured query model, visual authoring, backend query generation and an existing integration path. SonicGarden/blazer-querygen should not be forked as the SKY365 production base. It is a Ruby on Rails plugin for the unrelated Blazer BI product, not a Blazor/.NET component.

The useful combination is:

Natural-language request | v SKY365 Semantic Mapper / NL Query Planner | v Canonical SkyQueryPlan or EasyQuery-compatible Query JSON | v EasyQuery visual editor + human correction | v Policy + tenant + permissions + cost controls | v Provider compiler -> parameterized SQL / other target | v Preview -> execute -> audit

1. Verified SKY365 EasyQuery baseline

UI integrationQueryDesignerPanel.razor invokes sky365.initEasyQuery and returns both QueryJson and generated SQL.
Server integrationProgram.cs registers AddEasyQuery and maps EasyQuery endpoints.
Installed backend packagesKorzh.EasyQuery.* 7.4.1.
Client assetsApp.razor currently loads EasyQuery JavaScript/CSS version 7.2.3 from CDN.
Known defectThe backend/client minor-version mismatch can explain recorded API integration failures and must be resolved before declaring the designer production-ready.
Current upstreamNuGet lists Korzh.EasyQuery.AspNetCore 7.4.2, released 2026-04-09, as newer than the installed 7.4.1 package.

Required first action: align all backend packages, JS packages/CDN assets and product key to one tested EasyQuery minor version. Prefer a controlled 7.4.2 compatibility branch; do not upgrade production blindly.

2. Why EasyQuery validates the metadata-first direction

  • Its saved query model is JSON-based; SQL is an output, not the only authoring representation.
  • It separates data-model metadata from the user's query conditions, selected columns, sorting, grouping and aggregation.
  • It supports ASP.NET Core and frontends such as MVC/Razor, Angular, React and Vue. SKY365 currently hosts the JavaScript UI through Blazor interop.
  • It is suitable for advanced search, ad-hoc reporting, filtering, aggregation and grouping.

This supports the SKY365 design in which a Controller stores source object, fields, relationships, lookup displays and computed-field metadata, while a separate runtime query plan stores the user's current filter/group/sort/metric request.

Licensing boundary: EasyQuery is a commercial product governed by Korzh licensing. Public samples may be open source, but the core product must not be treated as a forkable MIT codebase unless the applicable source-code license has been purchased.

4. Blazer::Querygen pipeline

JavaScript injected into Blazer SQL editor -> POST prompt + current SQL + selected data source -> Rails PromptsController validates input lengths -> SchemaExtractor reads ActiveRecord schema -> PromptBuilder combines request + schema + current SQL -> AIClient calls OpenAI with retries -> QueryGenerator applies regex sanitization -> generated SQL is inserted into editor -> human reviews and executes through Blazer

The strongest design choice is that generated SQL is shown in the existing editor for review rather than executed automatically.

6. Material weaknesses

WeaknessWhy it matters to SKY365
No PK/FK relationship extractionThe prompt requests correct JOINs but provides tables, columns, types, comments and enums only. JOINs are inferred from names instead of governed relationship metadata.
Raw SQL is the AI outputThere is no canonical semantic plan, provider-independent representation or reliable round trip into a visual query model.
Regex sanitizerIt is not a SQL parser. It produces false positives for keywords inside comments/strings and does not constitute a complete read-only or side-effect safety boundary.
No tenant/row/column policyA syntactically SELECT-only query can still reveal sensitive data, cross tenants, scan huge tables or invoke provider-specific functions.
Data-source parameter not implementedThe extractor currently uses the default ActiveRecord connection even though the endpoint accepts a data-source name.
Arbitrary schema truncationIt takes the first configured maximum number of tables, rather than retrieving the entities and relationships relevant to the request.
Prompt loggingDebug logging records the full system and user prompt, including schema metadata/comments and current SQL. “No row data sent” does not mean no sensitive metadata is exposed or persisted.
No cost/explain gateNo join-depth, row, timeout, estimated cost or sensitive-field approval control is implemented by this plugin.

8. Proposed implementation contract

IQueryIntentPlanner Input: - natural-language request - optional existing SkyQueryPlan / EasyQuery Query JSON - scoped Semantic Catalog subset - user/tenant/profile context Output: - canonical SkyQueryPlan - ambiguities and confidence - requested fields/relations/metrics - risk classification IEasyQueryPlanAdapter - SkyQueryPlan -> EasyQuery Query JSON - EasyQuery Query JSON -> SkyQueryPlan - capability-loss report IQueryPolicyEvaluator - tenant predicates - row/column permissions - sensitive fields - max rows, joins, timeout, cost - approval requirement IQueryCompiler - parameterized SQL or another provider target - no direct execution of model-authored SQL

9. Immediate validation backlog

  1. Inventory every Korzh.EasyQuery.* package and every EasyQuery JS/CSS asset in the active solution.
  2. Create a branch aligning backend and frontend to 7.4.2, with license/product-key validation.
  3. Capture the real QueryJson generated by QueryDesignerPanel.razor.
  4. Test round trip: load JSON, edit condition, save JSON, regenerate SQL.
  5. Test a lookup displaying two and three concatenated fields.
  6. Test multi-condition joins using explicit AND/OR join predicates.
  7. Test computed fields, parameters, grouping, aggregation and aliases.
  8. Verify SQL dialect, parameterization, timeout and read-only enforcement.
  9. Add immutable tenant/security predicates that the visual editor and agent cannot remove.
  10. Only after the tests pass, decide whether EasyQuery Query JSON is the provider model beneath SkyQueryPlan or whether an adapter is required.
