Document, Image, Voice, Video and Media Engine Contracts

Evidence-separated assessment of SKY365 multimodal capabilities and plans.

Evidence boundary

The Knowledge Hub defines strategic direction and, for Video Builder, detailed renderer-neutral contracts. Searches of the inspected current implementation did not establish production OCR, image-analysis, speech or video-rendering runtimes. Screens, package-lock strings, planning documents and media files are not inference Engines.

1. Document Intelligence and OCR Engine

Status: documented/incubation; executable Engine not established.

Purpose
Extract verified structured information from Arabic/English business documents and connect it to governed actions.
Planned responsibilities
Intake, parsing/OCR routing, layout/table extraction, field confidence, provenance, review and evaluation.
Inputs → outputs
Document bytes + document type + tenant/policy context → text/layout/tables/typed fields, confidence and source coordinates.
Runtime
No owned OCR/extraction provider, pipeline or evaluated service found in the inspected current platform.
Metadata used
Future document type, extraction schema, field mapping, language, provenance and review policy. Existing document screens do not supply this contract.
Dependencies
File/Media lifecycle, provider adapters, Controller review screens, policy/audit, benchmark corpus and human review.
Extension points
Future OCR/parser adapters and document-type handlers.
Failure semantics
Must distinguish unreadable, unsupported, low-confidence, partial extraction and policy denial; none is implemented as a common contract.

Controller relation: a Controller can define target fields and review UI, but extracted evidence/provenance must remain attached to every proposed field update.

2. Image Intelligence Engine

Status: research/strategy; executable Engine not established.

Purpose
Analyze, classify or transform images under governed business workflows.
Inputs → outputs
Image asset + task/schema/policy → labels, detections, structured observations or generated derivative with confidence/provenance.
Runtime
No ImageAnalysis, computer-vision provider or owned inference boundary found.
Metadata used
No canonical image task/model/output schema found.
Dependencies
File/Media, consent/content safety, provider abstraction, evaluation and audit.
Extension points
Future analysis/generation provider adapters.
Failure semantics
Future contract must cover corrupt input, unsupported format, policy denial, low confidence, provider failure and unsafe output.

Controller relation: future image fields/actions may request analysis, but media assets and UI alone are not an Engine.

3. Voice Intelligence Engine

Status: research/strategy; executable Engine not established.

Purpose
Speech-to-text, text-to-speech, voice commands and governed call-agent interactions.
Inputs → outputs
Audio/text + language/voice/consent/context → transcript, speech asset, command intent or call event stream.
Runtime
No owned STT/TTS pipeline, call runtime or provider interface found in the current implementation scope.
Metadata used
No canonical voice, language, consent, diarization, command or retention schema found.
Dependencies
Media storage, streaming/provider adapters, identity/consent, policy, transcript provenance and evaluation.
Extension points
Future STT/TTS/telephony adapters.
Failure semantics
Future contract must cover no-speech, partial transcript, language mismatch, unsafe command, provider timeout and consent/retention denial.

Controller relation: voice commands must resolve to typed Controller actions and pass policy/approval, never execute from raw transcript text.

4. Video and Avatar Engine

Status: detailed documented architecture + interactive prototypes; production renderer not proven.

Purpose
Author, validate, preview, seek and render deterministic projects of scenes, layers, semantic objects, assets, animation, narration and avatars.
Responsibilities
Canonical project model, edit transactions, scene compilation, renderer adaptation, preview/seek, render jobs, progress/cancel, approvals and version history.
Inputs → outputs
Renderer-neutral VideoProject + validated edit operations → compiled project, preview/frame state, render job and output metadata.
Runtime
Knowledge Hub contains VB1–VB3 HTML prototypes and VB4–VB9 architecture. HyperFrames is accepted as first adapter, but the roadmap still requires an isolated spike, real MP4 and evidence report.
Metadata used
schemaVersion, projectId, title, canvas, assets, masterObjects, scenes, timeline, rendererConfig; scene/layer/instance/animation identities and typed edit scope.
Dependencies
Future scene compiler, asset freezing, renderer adapter, HyperFrames/Node/FFmpeg for the first implementation, approval/audit and render infrastructure.
Extension points
VideoRendererAdapter: validate, compile, preview, seek, renderScene, renderProject, progress and cancel. Renderer-specific values stay under adapter config.
Failure semantics
Unsupported properties must fail explicitly; compilation deterministic; preview/render share semantics; output records versions/checksums/warnings. These are documented requirements, not verified runtime behavior.

Controller relation: VideoProject has its own semantic model. Controllers can provide asset/project/approval screens, but must not replace scene/object identity or store renderer DOM selectors.

5. Media Asset Engine

Status: file storage primitives + planned video asset model; unified Engine not established.

Purpose
Own uploaded and generated assets, versions, provenance, access, transformations and lifecycle.
Existing responsibilities
FormBuilder local file save/get/delete/URL plus domain media/document screens.
Inputs → outputs
Uploaded file → local name/stream/URL; future frozen asset → content-addressed/versioned render input.
Runtime
Local FileStorageService under wwwroot/uploads; VideoProject assets are documented JSON, not a shared operational registry.
Metadata used
Current filename/path only; future asset ID, type, checksum, owner, source, version, rights, ACL and transformation metadata.
Dependencies
Storage providers, security/tenant policy, malware/content checks, audit and multimodal consumers.
Extension points
IFileStorageService is a local seam; no provider/asset lifecycle registry.
Failure semantics
Current component logs local IO errors; no common quarantine, retry, integrity, orphan cleanup or retention behavior.

Example and relationship

Controller upload/review screen → Media Asset identity → OCR/Image/Voice operation → confidence + provenance → human approval → typed Controller update; VideoProject → Scene Compiler → Renderer Adapter → preview/render evidence

The two flows share file, identity, policy, audit and evaluation needs, but they should not be collapsed into one generic “AI Engine.”

Future roadmap

  1. Establish Media Asset identity, ACL, provenance and checksums.
  2. Select one bilingual document workflow and benchmark it before Engine promotion.
  3. Define typed image/voice task contracts only with evaluation and consent rules.
  4. Complete Video Builder VB4 evidence: isolated adapter, deterministic seek, real output and report.
  5. Require a second renderer comparison at VB9 to verify portability.
  6. Expose all consequential multimodal outputs as drafts requiring validation/approval.
