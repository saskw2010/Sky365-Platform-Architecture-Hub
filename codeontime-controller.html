Code On Time Controller Architecture

Purpose: explain how legacy Code On Time Controller metadata becomes SKY365 screens and dynamic runtime behavior.

Finding

Code On Time Controllers are not merely legacy reference files. SKY365 contains an active bridge that discovers XML Controllers, analyzes and imports their metadata, converts it into runtime schemas and catalog entries, and renders the resulting Controllers through the dynamic UI and CRUD runtime.

Architecture and lifecycle

Code On Time dataController XML |-- fields / PK / types / lookup controllers |-- views / dataFields / categories `-- action groups / commands | |-- CodeOnTimeXmlParser -> CotDataController | `-- Sky365DynamicRuntimeService.MapCotToSchema | -> DynamicScreenSchema -> UI + CRUD + lookups | |-- ControllerXmlStudioService | -> analyze / format / save-with-backup / onboard | `-- SysControllerConfigService.ImportFromLegacyXmlAsync -> SysController + SysControllerFields -> UILayoutJson + ActionsJson + versions Onboarding -> module schema + module catalog -> unified catalog -> runtime URL / preview URL / dynamic API

Runtime components and extension points

  • CodeOnTimeXmlParser searches web controllers, Salogi and Noor Controller folders and deserializes XML.
  • ControllerXmlStudioService finds module Controller folders, validates XML paths, analyzes fields/views/lookups, saves with backup and delegates import to onboarding.
  • BatchControllerImporterService copies XML into module ownership and derives module keys from Controller prefixes.
  • SysControllerConfigService supports database, JSON and legacy XML imports plus version snapshots.
  • Sky365DynamicRuntimeService maps parsed Code On Time objects into the same runtime schema consumed by dynamic screens and CRUD.

Deviations and gaps

  • There are two related persistence models: file-based dynamic schemas/catalogs and database-backed System.SysControllers. Their canonical ownership and synchronization contract are not yet unified.
  • The lightweight XML parser and Studio importer do not necessarily preserve every Code On Time construct. A field-by-field loss matrix is required.
  • Some generated JSON contains historical absolute xmlPath values. Machine-specific paths are provenance, not portable runtime dependencies.
  • Repository counts do not prove every generated screen loads successfully; runtime sampling is a separate gate.

Future roadmap

Create a canonical Controller Intermediate Representation, record lossless source provenance, compile XML and authored JSON into it, then validate generated UI, CRUD and API behavior with golden Controllers covering lookup, master-detail, actions, permissions and workflows.
