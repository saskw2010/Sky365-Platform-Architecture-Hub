Code On Time Controller Conversion Fidelity

Purpose: show exactly which source constructs are preserved by each current import/runtime path.

Finding

SKY365 has three active Code On Time conversion paths. They do not preserve the same metadata. Therefore the selected import path can change the generated screen even when the source XML is identical.

Three paths

A. Runtime fallback parser

CodeOnTimeXmlParser deserializes a deliberately small CotDataController model. MapCotToSchema maps fields, coarse primitive types and simple lookups. It ignores parsed views and guesses lookup key/text fields.

B. Dynamic onboarding

DynamicControllerOnboardingService parses XML using LINQ to XML, writes a generated dynamic JSON schema, retains the XML, upserts the module catalog and synchronizes the unified catalog. It captures more field state but does not convert views/actions/workflow.

C. Database-backed Studio import

SysControllerConfigService.ImportFromLegacyXmlAsync creates System.SysControllers and field rows, plus JSON UI layout/actions. It preserves more presentation/action intent but feeds a distinct persistence model.

Architectural risk

  • Two users importing the same XML through different screens can receive different metadata fidelity.
  • A successful import does not indicate lossless conversion.
  • Guessed table, key and lookup fields can produce a screen that renders but queries incorrect data.
  • Missing views/actions/rules can turn an operational Controller into a generic CRUD screen.
  • Separate file and database Controller models can drift without an explicit synchronization owner.

Required verification

Create golden XML Controllers covering primary key, lookup, master-detail, multiple views, categories, actions, rules and permissions. Compile each through every path, compare normalized CIR output and run non-mutating screen/API probes.
