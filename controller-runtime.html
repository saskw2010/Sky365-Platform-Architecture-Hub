Controller Runtime

Purpose: explain the verified path from authored metadata to observable runtime behavior.

Runtime lifecycle

Root systems registry → system module registries → module Controller catalogs → unified catalog (merge + normalize + deduplicate) → resolve Controller + module + system + profile → load JSON into DynamicScreenSchema ├─ UI: fields / sections / views / actions ├─ CRUD: query / create / update / delete ├─ Lookups: primary / fallback sources ├─ Rules: pre-save / post-save / UI actions ├─ Security: roles / module authorization ├─ Workflow: embedded or linked definition └─ Tools: describe / list / get records

Central runtime object

DynamicScreenSchema contains table identity, fields, detail schemas, sections, views, pre-save rules and post-save actions. Fields contain lookup and many-to-many definitions; views contain actions and fields. Sky365DynamicRuntimeService.LoadSchemaAsync resolves a scoped schema and materializes that object. Its CRUD, lookup, authorization and action methods consume the result.

Relationships and deviations

A Controller/catalog entry locates and describes a schema, but the evidence does not support saying that one Controller JSON directly generates every subsystem. Workflows, dashboards and reports can be separately ingested and linked by MetadataGraphService. AI tools adapt the catalog/runtime; they are consumers rather than proof of an AI-native Controller compiler.

Capability matrix

CapabilityRelationship to Controller metadata
UI / CRUD / lookup / validationDirect and verified through DynamicScreenSchema and runtime/UI services.
API / search / permissionsRuntime-mediated and verified in part; endpoint composition and policy ownership need deeper trace.
Workflow / dashboard / reportsLinked metadata domains; not proven to be wholly embedded in one Controller.
Worker / schedulerNot established in this baseline.
