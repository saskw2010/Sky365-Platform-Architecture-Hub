Agentic, Knowledge and Trust Engines

Reference for implemented and partially implemented agent-facing runtime boundaries.

Current relationship

User / Screen / Studio shell -> Skill Router or Unified/Mascot Agent -> Shared Action Core V1 |-- deterministic route |-- local knowledge grounding |-- Dynamic Controller read tools |-- draft-only workflow artifact |-- policy / approval decision `-- trace + durable audit No general autonomous mutation runtime is proven.

1. Shared Agent Runtime

Maturity: partial and distributed.

Purpose: receive conversational/business requests and coordinate intent, context, skills, tools, answers and safe workflow output.

Responsibilities: request context, deterministic/AI-assisted intent, route selection, skill/tool use, validation and response assembly.

Inputs: question, user/tenant/role, page/system/module context and optional trace ID. Outputs: answer, intent, route, citations, tool calls, draft actions, workflow steps and handoff/approval flags.

Runtime evidence: Unified and Mascot controllers/services, SkillRouter, Action Core and trace pipeline. Metadata: request context, skill definitions, Controller catalog and route hints.

Dependencies: Action Core, Dynamic Tools, AI service, policy/audit and conversation history. Extension points: skills, routes, tools and shells.

Gap: there is no single versioned agent-runtime contract covering lifecycle, cancellation, recovery, state and tool governance.

2. Shared Action Core V1

Maturity: implemented reference subsystem, deliberately narrow.

Purpose: route selected business requests through a safe, auditable, draft-first path.

Responsibilities: normalize input; choose deterministic route; handle Studio generation assistance, supplier approval workflow and Dynamic Metadata reads; persist drafts; ground knowledge; audit; reject unsupported work safely.

Inputs: ActionRequestDto. Outputs: ActionCoreResponse with route plan, outcome, steps, metadata, citations and suggested/draft actions.

Runtime: ActionCoreService.HandleAsync and ActionRoutePlanner. Metadata: source shell, session/trace, Controller context and route scenario.

Dependencies: Dynamic Tool Gateway, knowledge grounding, draft persistence, AI service, audit and observability. Extension points: supported scenario handlers and route decisions.

Important boundary: source explicitly states only selected V1 scenarios are enabled; unsupported requests execute no business mutation.

3. Dynamic Tools and MCP Gateway

Maturity: advanced partial.

Purpose: expose authorized Controller discovery and read operations to agents and MCP-style consumers.

Responsibilities: list Controllers; describe schema; list/get records; resolve system group; enforce roles; log database/tool calls; provide MCP prompts/tools and HTTP endpoints.

Inputs: user principal, Controller/search/system scope and record ID. Outputs: tool-safe Controller/schema/record contracts.

Runtime: DynamicToolGateway, DynamicMcpTools, DynamicMcpPrompts, Dynamic Tools controller and MCP Lab.

Metadata: unified catalog and DynamicScreenSchema. Dependencies: Dynamic Runtime, catalog, authorization and trace services.

Extension points: new read-only tools and controlled action tools. Gap: complete MCP protocol lifecycle, tool versioning and mutation approval contract are not yet proven.

4. Knowledge Grounding Engine V1

Maturity: implemented local keyword grounding; not Vector RAG.

Purpose: attach small policy-corpus evidence to selected workflows.

Responsibilities: load four Markdown policy documents, split them by headings, cache snippets, tokenize query/intent, score title/section/body matches and return citations/warnings.

Inputs: query, intent, role, tenant and maximum results. Outputs: matched KnowledgeSourceSnippet items, scores, references and grounding metadata.

Runtime: KnowledgeGroundingService, mode KnowledgeLaneV1. Metadata: Markdown headings, source IDs, references and request context.

Dependencies: filesystem and in-memory cache. Extension points: corpus sources, tokenizer and scorer.

Gap: no embeddings, vector database, hybrid retrieval, reranker, document ACL model or evaluation suite is evidenced.

5. Policy and Approval Engine V1

Maturity: implemented deterministic evaluator, not a general declarative policy engine.

Purpose: map agent response state to risk, permission and approval outcomes.

Responsibilities: derive Low/Medium/High risk; Allowed/AllowedWithApproval/Blocked; route hints; approval reasons; DraftOnly/human-handoff behavior; policy trace output.

Inputs: request, user/tenant context and assembled response flags. Outputs: AgentPolicyDecisionSnapshot.

Runtime: MascotPolicyEngineService plus Action Core draft controls. Metadata: currently code-defined flags and routes, not external policy documents.

Dependencies: agent response and context. Extension points: policy rules and permission providers.

Gap: policy-as-data, policy versions, conflict resolution, tenant overrides and cross-engine enforcement are missing.

6. Flight Recorder, Audit and Observatory

Maturity: advanced partial.

Purpose: make agent decisions and side effects inspectable.

Responsibilities: record context, intent, policy, selected route, action plan, validation, entities, steps, tool calls, DB calls, LLM calls, memory and policy traces; query summaries/details; persist mascot audit; expose observability dashboards and metrics.

Inputs: request/context/response and execution events. Outputs: persistent trace graph, audit records, metrics and dashboard data.

Runtime: AgentTraceService, PersistentMascotAuditService, Agent Observatory contracts/controller/dashboard.

Metadata: trace ID, tenant/user/session, route, risk, tool/data calls, timings and serialized input/output.

Dependencies: application database and instrumented callers. Extension points: trace interfaces for tools, DB, LLM, memory and policy.

Gap: cross-agent trace standards, alerting, replay, retention/redaction and outcome linkage are not yet fully evidenced.

7. Skill Engine

Maturity: implemented runtime subsystem; packaging governance partial.

Purpose: route user messages into reusable Controller-aware behaviors.

Responsibilities: register/find skills; route requests; discover/open screens; search/open records; optionally bridge actionable requests to Action Core.

Inputs: SkillRequest and user context. Outputs: AiResponse with buttons, routes, records or Action Core response.

Runtime: SkillRegistry, SkillRouter, DiscoverScreensSkill, OpenScreenSkill, SearchRecordsSkill, OpenRecordSkill.

Metadata: SkillDefinition names/descriptions/tools. Dependencies: Dynamic Tool Gateway and Action Core.

Extension points: implement ISkill. Gap: versioned packages, permissions, dependencies, test contract and external distribution are not established.

8. Agent Evaluation Ledger

Maturity: early partial.

Purpose: persist reproducible agent test runs and assertions.

Responsibilities: start/complete runs; record steps, tool calls, assertions and artifacts.

Inputs: scenario key/input and observed execution data. Outputs: test ledger records with status, trace and artifacts.

Runtime: AgentTestLedgerService and Agent Test Runs controller. Dependencies: application database.

Extension points: new assertions and artifact types. Gap: curated evaluation packs, baseline comparison, CI gates, safety/red-team suites and scoring policy.

Controller and screen relationship

Agentic capabilities currently consume Controllers primarily through read-only discovery, schema description, record search and screen navigation. This is a strong MDAP integration point: Controller metadata becomes both human UI and agent tool context. It is not yet proof that Controller metadata defines agent policies, prompts, workflows or write tools end-to-end.
