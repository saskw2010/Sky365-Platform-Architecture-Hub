Legacy Code On Time Runtime Architecture

Purpose: explain what a full legacy Controller controlled before SKY365's modern dynamic runtime.

Legacy Controller breadth

A representative legacy dataController includes SQL commands, field definitions, Grid/Edit/Create views, categories, data fields and multiple action groups for selection, CRUD, duplicate, import/export, datasheet/grid modes and reports. This is substantially richer than the current lightweight fallback CotDataController model.

Legacy runtime relationship

Data Aquarium project configuration -> database discovery / code generation -> dataController XML |-- commands and SQL projection |-- fields and relationships |-- grid/edit/create views and categories |-- action groups and conditional commands `-- report/export/import behavior | v Generated .NET Framework runtime |-- DataControllerBase / factory / filters |-- BusinessRules / ActionHandler |-- ApplicationServices / membership |-- REST services `-- Report handlers + Touch UI

Migration implications

  • The legacy XML source contract is richer than CotModels.cs; parsing fields and view IDs alone cannot preserve behavior.
  • Commands/SQL, categories, conditional action attributes and report/export actions need explicit CIR representation or declared unsupported status.
  • Legacy generated framework code should be studied for semantics, not copied wholesale into the modern runtime.
  • Modern metadata adds profiles, lifecycle rules, workflow links, graph identities, tools and semantic opportunities not native to the legacy model.

Example and roadmap

A golden legacy Controller should include one query command, primary key, lookup, Grid/Edit/Create views, categories and all major action scopes. Its conversion report must show which behavior is preserved in VGem3, API, reports and tools.
