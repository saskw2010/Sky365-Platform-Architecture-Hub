Classification rule

A suffix is not architecture. *Service, *Builder or *Engine records an implementation name; promotion to a platform Engine requires an owned contract, metadata and runtime lifecycle. Counts below describe public classes in Sky365Offers.BlazorWeb/Features/DynamicEngine at the pinned revision.

Background work and missing compiler boundary

  • MetadataGraphWarmupService : IHostedService performs startup warm-up.
  • ProviderHealthMonitor : BackgroundService exists under Studio rather than DynamicEngine and monitors providers.
  • Neither supplies common job metadata, queues, leases, retries, idempotency or execution history; therefore no general Worker Engine is claimed.
  • No *Compiler public class exists in DynamicEngine. Parser, onboarding, scaffolding, catalog normalization and runtime loading currently distribute compiler-like responsibilities.

Controller and screen relationship

XML / JSON / registries → discovery + onboarding + schema services → catalog identity → runtime + UI + persistence → workflow/dashboard/report/integration → graph/tools/observability

Code On Time Controllers enter through parser/onboarding/import paths. Their semantics can then be rendered or executed by multiple runtime components. Because compilation is distributed, no single component currently guarantees end-to-end preservation.

Roadmap

  1. Assign explicit owners to parser, normalization, validation and compiled output.
  2. Define a Controller compiler contract and diagnostics without assuming implementation.
  3. Keep page-level Builders distinct from metadata compilers.
  4. Introduce Worker/Scheduler contracts only when durable failure semantics are defined.
