Living Architecture Knowledge Governance

Repeatable process for refreshing source evidence without erasing prior findings.

Operating rule

Source revision → inventory → claim/evidence comparison → page update → ADR if decision changes → verification → discovery ledger

The Architecture Hub reconciles evidence. It does not copy production source, credentials or unverified strategy claims into implementation status.

Evidence refresh procedure

  1. Record repository URL, branch, full commit SHA, inspection date and dirty-worktree state.
  2. Preserve the prior finding; change status only when stronger source/runtime evidence exists.
  3. Re-run Controller, screen, schema and Engine inventories using the same counting definition.
  4. Compare added, removed and changed identities. Never compare only display names when structural matching is required.
  5. Trace new claims from Knowledge Hub vocabulary to executable source and runtime paths.
  6. Update the catalog, maturity matrix, property reference and affected diagrams together.
  7. Create or supersede an ADR when identity, ownership, lifecycle, trust boundary or evidence authority changes.
  8. Run local-link, required-capability and source-reference checks.
  9. Append a discovery-ledger entry describing evidence and impact.

Status transition rules

TransitionMinimum evidence
Unverified → DocumentedOwned blueprint/ADR/roadmap at a recorded revision.
Documented → PrototypeExecutable demonstrator with identifiable scope and limitations.
Prototype → PartialReusable executable components in a real platform path.
Partial → Implemented EngineOwned boundary, contract, metadata, inputs/outputs, dependencies, extension/failure semantics and tests.
Implemented → Operationally verifiedRuntime/deployment evidence, telemetry and failure-path verification.
Any status → Regressed/unknownMissing source, broken runtime evidence, removed owner or incompatible revision; uncertainty is recorded, not hidden.

Counting definitions

  • JSON files under Schemas: filesystem artifacts; may include registries/governance and are not automatically screens.
  • Catalog screens: normalized catalog items whose node type is Screen.
  • Code On Time XML Controller: XML document containing a dataController root/element; generated copies are classified separately.
  • Engine: owned runtime boundary; a class name, page or strategy label alone is insufficient.
  • Implemented experience: a real portal/prototype may be implemented without being a platform Engine.

Change and supersession

  • Extend pages or add new versions; do not remove contrary historical evidence without explanation.
  • Every page states evidence scope and avoids global absence claims from a limited repository set.
  • Secrets are never copied. Only the risk, remediation requirement and source category are recorded.
  • Generated counts include the command definition or counting rule in the discovery ledger.
  • Conflicting pages are corrected in the same change and cross-linked to the new evidence.
